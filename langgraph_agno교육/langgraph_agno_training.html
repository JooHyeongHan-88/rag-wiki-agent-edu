Section 1

1Tool Calling의 한계와 Agent Framework의 등장

에이전트 시스템을 제대로 이해하려면 먼저 ChatGPT나 Claude 같은 LLM의 도구 호출(Tool Calling)의 근본적 약점부터 알아야 합니다. Agent Framework는 바로 이 약점을 딛고 탄생했습니다.

1-1. 관통 예시로 한계를 체감하기

SaaS 고객 지원 시스템인 support-bot에 단순한 질문이 접수됩니다.

🧪 실무 시나리오 — Level 1: "배송 어디쯤 왔나요?"

이건 단순한 Tool Calling 한 번이면 해결됩니다. LLM이 track_order(order_id="1234")를 호출하라고 판단하면, 애플리케이션 코드가 이를 실행하고 결과를 LLM에 건네줍니다. LLM은 "배송 중입니다"라고 고객에게 예쁘게 답변합니다. 여기선 프레임워크가 필요 없습니다.

하지만 고객의 요구사항이 조금만 복잡해져도 사정은 완전히 달라집니다.

🧪 실무 시나리오 — Level 2: "주문 1234번 환불해 주세요."

이를 안전하게 처리하려면 시스템 내부에서 다음과 같은 일련의 과정이 순차적으로, 조건에 따라 일어나야 합니다.

  1. 의도 분류 (Intent Classification): "환불 요청"인지 식별
  2. 주문 조회 (Order Retrieval): track_order("1234") 호출 → 배송 상태, 결제액, 구매자 대조
  3. 환불 정책 확인 (RAG): 회사 규정 문서를 검색(RAG)해 이 제품이 환불 기간 내에 있는지 판정
  4. 결제 수단 조회 (Payment Method Lookup): 신용카드인지, 네이버페이인지 조회
  5. 금액별 승인 분기 (Conditional Routing): 50만 원 초과면 사람 승인 대기 / 이하면 자동 처리
  6. 환불 실행 (Refund Execution): issue_refund("1234") API 호출
  7. 기록 및 안내 (Notification): 고객 DB에 이력을 남기고 최종 결과를 친절하게 말로 작성

이 다단계 흐름을 "LLM에게 도구 7개를 다 쥐어주고 알아서 하라"고만 방임하면, 현업 프로덕션에서는 예외 없이 다음 대참사가 발생합니다.

  • LLM이 정책 확인(RAG) 단계를 멋대로 건너뛰고 환불 API부터 바로 때린다.
  • 50만 원이 넘는 고액 환불을 사람 승인 없이 승인 판정을 내고 돈을 보낸다.
  • 4단계에서 결제 API 타임아웃이 났을 때, 재시도(Retry)를 안 하거나 예외 처리를 못 해 전체 시스템이 뻗는다.
  • 똑같은 도구를 무한 호출하며 루프에 빠져 토큰 요금 폭탄을 청구한다.
그림 1. support-bot의 업무 흐름 점층화와 요구되는 에이전트 오케스트레이션 계층

1-2. 그래서 직접 짜 보면 — "오케스트레이션 코드"의 탄생

LLM 단독 실행에 맡기지 않고, 파이썬 코드로 전체 흐름을 장악하려 할 때 가장 소박하게 작성할 수 있는 기본 에이전트 루프입니다.

Python · 의사코드 
# 💻 의사코드 (구조/원리) — Tool Calling을 직접 엮은 가장 단순한 Agent
def naive_agent(user_message, tools):
    messages = [system_prompt, user_message]
    while True:
        response = llm(messages, tools=tools)        # 1) LLM이 상황을 보고 '생각'
        if response.tool_calls:                       # 2) 도구를 쓰겠다고 요청하면
            for call in response.tool_calls:
                result = tools[call.name](**call.args)  # 3) 애플리케이션 단에서 함수 실행
                messages.append(tool_result(result))    # 4) 실행 결과를 대화 이력에 추가
            continue                                  # 5) 결과를 들고 다시 루프를 돌아 LLM에게 전달
        return response.content                       # 6) 도구 호출이 더 없으면 최종 답변 반환

이 단순한 while 루프가 사실상 에이전트의 정수(Essence) 입니다. LLM이 스스로 "도구를 더 쓸지, 여기서 멈추고 답을 낼지"를 판단하는 주체로 작동하기 때문입니다.

하지만 이 20줄짜리 루프를 진짜 현업 프로덕션급 support-bot으로 키우려고 하면 다음과 같은 요구사항들이 빗발치며 배관 코드가 감당할 수 없을 정도로 복잡해집니다.

프로덕션 요구사항 필요성 바닐라 파이썬으로 구현 시 겪는 고통
조건 분기 및 복귀 50만 원 초과 건은 무조건 상담원 승인으로 분기 if/else 조건절이 while 루프 안에 거미줄처럼 뒤엉켜 스파게티 코드가 됨
공유 상태 관리 (State) 인증 정보, RAG 검색문서, 의사결정 결과를 단계 간에 정확히 유지 전역 dict를 노드들이 여기저기서 수정하다가 데이터 경합 및 추적 불가 상태 돌입
사람 개입 (HITL) 상담원이 승인 버튼을 누르기 전까지 대기 상태로 정지 while 루프 중간을 강제로 일시정지(Suspend)하고 프로세스를 죽였다가 며칠 뒤 그 자리에서 그대로 복원(Resume)하는 코드를 짜야 함
에러 복구 (Retry) 타사 결제사 API 네트워크 실패 시 대응 루프 전체를 감싸는 try/except 처리와 재시도 횟수 상태 추적 코드로 코드가 뚱뚱해짐
비정상 루프 차단 동일 도구를 바보같이 무한 호출하는 토큰 소모 차단 도구 호출 횟수를 계수(Counting)하고 강제 탈출하는 안전 가드를 매 루프마다 배치해야 함
비동기/병렬 (Parallel) 자금 확인과 재고 조회를 동시에 진행 asyncio나 스레드를 직접 다뤄 조율하는 고난도 동시성 배배 꼬인 코드가 추가됨
영속 실행 (Durability) 환불 처리 중에 웹 서버가 급작스럽게 종료됨 모든 while 상태를 데이터베이스에 단계마다 체크포인트로 직렬화(Serialize)해 보관해야 함
💡 비유 — 1인 셰프 식당 vs 대형 레스토랑 주방

LLM 자체는 솜씨 좋은 셰프입니다. 식재료(컨텍스트)를 얹어주면 훌륭한 요리(답변)를 냅니다. Tool Calling은 셰프가 보조원에게 "소금 통 좀 가져다줘"라고 외치는 능력입니다.

하지만 하루에 1,000명의 손님이 드나드는 대형 레스토랑(프로덕션)을 운영하려면 셰프 한 명의 요리 실력만으론 불가능합니다. 주문 순서를 교통정리하고(제어 흐름), 재료 소진을 추적하며(상태 관리), 고가 메뉴 주문은 점장 확인을 거치고(사람 개입), 사고가 나면 대피시키는(에러 처리) 주방의 약속된 시스템이 필요합니다. Agent Framework가 바로 이 레스토랑의 주방 시스템입니다.

📌 에이전트 프레임워크가 푸는 핵심은 "LLM 호출 자체"가 아니라, 그 주변을 싸고도는 복잡한 오케스트레이션(제어 배관)입니다.

1-3. 두 갈래의 답 — "그래프로 통제" vs "Agent 객체로 추상화"

이 지독한 오케스트레이션 문제를 대하는 현대 프레임워크의 두 흐름이 바로 이 교육의 주인공입니다.

🛡️ LangGraph의 답: "그래프로 통제"

에이전트가 돌 흐름도를 명시적인 그래프(Node와 Edge)로 개발자가 직접 철저하게 조작하게 하자. LLM이 마음대로 날뛰지 않도록 엄격한 궤도(통제권)를 제공하겠다는 철학입니다.

⚡ Agno의 답: "Agent 객체로 조립"

배관 코드를 개발자에게 노출하지 말고, 기억·도구·지식을 인자로 꽂는 완성형 Agent 객체를 주어 조립하게 하자. 빠른 구현과 개발자 생산성에 올인하겠다는 철학입니다.

단일 도구 호출을 넘어, 다단계 분기·상태 관리·사람 개입·예외 복구가 엮인 실제 에이전트 시스템을 짜려면 복잡한 배관 코드가 필요한데, 이를 그래프로 명시해 통제(LangGraph)할 것인지 완성형 객체로 선언해 생산성을 챙길 것인지(Agno)가 프레임워크를 가르는 두 갈래 길이다.

Section 2

2Agent 시스템의 공통 어휘 8개

LangGraph와 Agno는 서로 다른 인터페이스를 가지지만, 본질적으로 에이전트 시스템을 만들기 위해 같은 8가지 핵심 단어를 사용합니다. 이 공통 약속을 먼저 알면 두 프레임워크의 코드가 1:1로 겹쳐 보이기 시작합니다.

2-1. Workflow와 Agent — 자율성의 스펙트럼

개발자들이 가장 많이 혼동하고, 잘못 섞어 쓰는 단어입니다.

  • Workflow (워크플로): 다음 단계가 무엇인지 애플리케이션 코드(if/else, 정적 엣지)가 미리 강제한 구조입니다. LLM은 각 칸 안에서 텍스트를 정리하거나 변환하는 요리사로만 쓰입니다. (결정론적, 예측 가능성)
  • Agent (에이전트): 다음 행동을 LLM이 런타임에 유연하게 직접 계획하고 결정합니다. 지금 답변을 보낼지, 다른 도구를 하나 더 열어볼지 주도적으로 처리합니다. (비결정론적, 유연성)
결정론적 (Deterministic) 자율적 (Autonomous)
Workflow (코드가 운전대 장악) Agent (LLM이 운전, 코드는 조수석)
⚠️ 함정 — 100% 자율적인 순수 Agent는 프로덕션에 없다

현업에 뛰어드는 초심자들은 "모든 것을 LLM이 혼자 계획하고 도구를 찾아 쓰는 자율 에이전트"를 꿈꿉니다. 그러나 이는 통제가 불가능하고 무한 루프에 약해 상용 제품엔 쓰기 곤란합니다. 실제 성공하는 프로덕션은 '굵은 뼈대는 Workflow(코드)로 가이드라인을 깔고, 지엽적이고 유연한 소통 영역만 Agent(LLM)에게 위임하는 하이브리드 형태'를 취합니다.

2-2. State — 흐름을 가로지르는 "공유 차트"

에이전트가 여러 노드를 건너다니는 동안 공통적으로 읽고 쓰는 작업 메모리(Context)입니다.

💡 비유 — 환자의 병원 차트

환자(고객 요청)가 내원하면 [접수대 ➔ 진료실 ➔ 검사실 ➔ 처방전]을 차례로 돕니다. 접수원, 의사, 임상병리사는 각자 자기 역할만 처리하지만, 환자가 겨드랑이에 끼고 다니는 하나의 병원 차트에 정보를 기록하고 앞 단계 기록을 읽어들입니다. 이 차트가 에이전트 시스템의 State입니다.

LangGraph는 이 State를 개발자가 데이터 스키마로 직접 엄격하게 짜도록 유도하고, Agno는 세션 구조 안에 숨겨 프레임워크가 알아서 관리해 줍니다.

2-3. Memory — 세션을 넘어선 기억

State가 "현재 하나의 환불을 처리하는 동안만 유지되는 임시 차트"라면, Memory는 통화가 종료되고 다음 주에 고객이 다시 찾아왔을 때도 기억되는 정보입니다. 보통 단기 기억(동일 대화 세션 내 맥락 유지)과 장기 기억(세션을 초월해 사용자의 성향, VIP 여부, 과거 불만 사항 누적)으로 구분하여 보관합니다.

2-4. Tool — 외부 세계를 움직이는 손발

LLM 자체는 뇌(텍스트 예측기)에 불과합니다. 에이전트가 메일을 발송하거나, 데이터베이스를 업데이트하거나, 사내 규정 PDF에서 정보를 추출(RAG)할 수 있도록 연결된 파이썬 함수들이 바로 Tool입니다. RAG는 에이전트 시스템 전체가 아니라, 단지 에이전트가 지식을 얻기 위해 쓸 수 있는 '하나의 특수한 도구'일 뿐입니다.

2-5. Routing — 갈림길의 안내자

"이 질문은 환불 부서로 갈 것인가, Q&A 부서로 갈 것인가?"를 판단해 다음 노드로 프로세스를 밀어주는 행위입니다. LangGraph는 이를 Conditional Edge로, Agno는 내부 ReAct 의도 파악으로 해결합니다.

2-6. Multi-Agent — 전문가들의 분업

한 명의 대가 LLM에게 환불, 추천, 기술 조치, DB 정리 등 50개의 도구를 다 주고 혼자 다 하라고 시키면 프롬프트도 무거워지고 오동작 확률이 치솟습니다. 대신 '환불 담당', '추천 담당', '기술지원 담당'으로 아주 잘 훈련된 소형 에이전트들을 각각 두고, 이들이 대화를 나눠가며 협업하게 쪼갭니다. 이를 멀티 에이전트 아키텍처라 합니다.

2-7. Orchestration — 조율자

멀티 에이전트들을 나열해 놨을 때, "고객의 첫 마디를 누가 처리하고, 다음 업무는 누구에게 토스할지" 교통정리를 해주는 총감독의 역할입니다. (리더형, 순차 배달형, 난상 토론형이 존재합니다.)

🧭 Section 2 한 줄 요약

에이전트 시스템은 Workflow/Agent(자율성) · State(공유 정보) · Memory(기억) · Tool(도구) · Routing(분기) · Multi-Agent(분업) · Orchestration(조율)이라는 8가지 개념으로 굴러가며, 이 중 State를 개발자가 설계하느냐 프레임워크가 숨겨주느냐가 양강의 갈림길이다.

Section 3

3설계 철학 — "Agent를 그래프로"

LangGraph는 LangChain 팀이 직접 만들었습니다. 하지만 그 배경에는 초기 LangChain의 고수준 추상화인 Chain(체인)의 한계에 대한 치열한 반성이 서려 있습니다.

3-1. 무엇에 반발해서 태어났나

초기 LangChain의 핵심 모델인 Chain은 "프롬프트 ➔ LLM ➔ 파서"로 이어지는 선형 파이프라인에 최적화되어 있었습니다. 그러나 현실의 진짜 에이전트는 한쪽 방향으로만 달리지 않습니다.

  • 순환(Loop)이 기본이다: 에이전트는 "생각하고 ➔ 도구를 부르고 ➔ 결과를 수거해 ➔ 다시 생각하는" 제자리 돌기(순환)를 무조건 필요로 합니다.
  • 비결정론적 상태 공유: 여러 도구와 노드가 동일한 역사(대화 이력)와 정보(State)를 뒤엉키지 않게 계속해서 액세스하고 고쳐 쓸 수 있어야 합니다.
  • 제어 가시성 상실: 기존의 AgentExecutor 같은 고수준 마법 객체들은 루프 처리를 블랙박스 안에 꽁꽁 싸매놓아, 개발자가 "도대체 왜 여기서 엉뚱한 도구를 계속 호출하는지" 세부 엣지를 튜닝하거나 간섭하기가 불가능했습니다.

📌 그래서 탄생한 LangGraph의 철학은 "에이전트의 제어권을 숨기지 말고, 개발자 눈앞에 그래프 설계도로 완전히 까뒤집어 보여주자"입니다.

3-2. 왜 하필 "그래프"인가

선형 리스트는 루프를 표현할 수 없고, 트리 구조는 갈래는 칠지언정 다시 하나의 노드로 합쳐지거나 뒤로 돌지 못합니다. 순환, 분기, 그리고 여러 갈래의 합류를 수학적으로 모두 보장하는 가장 보편적인 구조가 바로 그래프(Graph)입니다.

💡 비유 — 지하철 노선도와 열차

LangGraph의 에이전트는 지하철 노선도와 같습니다. 역(Node)이 있고 역과 역을 잇는 철로(Edge)가 있습니다. 그리고 승객들의 상태 정보가 담긴 열차(State)가 선로 위를 따라 달립니다.

노선을 짜는 사람은 선로를 정해두어 열차가 절대 엉뚱한 탈선 사고를 내지 못하도록 강제합니다. 컴퓨터과학에서 말하는 상태 기계(State Machine)의 원리를 LLM 오케스트레이션에 고스란히 이식한 것입니다.

3-3. LangGraph의 핵심 철학 세 가지

핵심 철학 의미 개발자에게 주는 실질적 가치
Low-level & Explicit
(저수준과 명시성)		 마법처럼 알아서 해주는 기능을 최소화하고, 모든 상태 전이와 노드 연결을 손수 선언하게 합니다. 흐름의 완전한 장악. 탈선하지 않는 고신뢰성 에이전트 구축 가능.
State-first
(상태 최우선)		 전역 작업 테이블인 State를 중심에 세우고, 모든 노드는 오직 이 State를 읽고 쓰며 소통합니다. 구조의 투명성. 중간 단계마다 어떤 데이터가 입출력되는지 한눈에 디버깅.
Durable & Inspectable
(내구성과 관찰 가능성)		 매 동작 단계마다 데이터를 디스크/메모리에 영구 저장하여 상태를 영속화합니다. 장기 실행 지원. 서버가 다운되어도 복귀하며, 과거로 되돌아가는 시간 여행(Time-travel) 가능.

LangGraph는 에이전트의 루프와 오케스트레이션을 블랙박스에 감추지 않고, 개발자가 직접 그래프(상태 기계)로 그려 세부 통제권을 가져가게 설계된 명시성 중심의 프레임워크다.

Section 4

4Node · Edge · State — LangGraph의 빌딩 블록

LangGraph를 짜기 위해 우리가 만지는 재료는 딱 세 개입니다. 데이터 설계도(State), 일꾼(Node), 길잡이(Edge). 이 셋의 선언이 LangGraph 프로그래밍의 처음과 끝입니다.

4-1. State — 모든 칸이 공유하는 단 하나의 차트

노드들이 공유하며 갱신하는 공통 구조체입니다. LangGraph에서는 주로 파이썬의 TypedDict로 명시적 설계도를 정의합니다.

Python · LangGraph 
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

# 💻 의사코드 (구조/원리) — State는 "이 시스템의 작업 메모리 설계도"
class SupportState(TypedDict):
    messages: Annotated[list, add_messages]  # reducer: 기존 메세지에 '누적'
    order_id: str | None                     # 덮어쓰기 (기존 값 교체)
    intent: str | None                       # 덮어쓰기
    policy_ok: bool                          # 덮어쓰기
    amount: int                              # 덮어쓰기
    approved: bool                           # 덮어쓰기

📌 Annotated[list, add_messages] 처럼 특정 필드에 Reducer(리듀서) 함수를 지정할 수 있습니다. 노드가 새 값을 돌려줄 때, 기본값은 기존 값을 밀어내고 덮어쓰는 것이지만, Reducer가 지정되면 지정된 규칙(여기선 기존 대화 배열 뒤에 추가)에 따라 상태가 병합됩니다.

4-2. Node — 묵묵히 일하는 노드 함수

노드는 실제 로직(LLM 호출, API 검색, DB 수정 등)을 품은 파이썬 함수입니다. 규칙은 단 하나입니다.

📌 Node 함수의 입력과 출력 규칙

노드는 현재 State를 인자로 받아, '자기가 수정한 일부 데이터가 담긴 dict'를 반환합니다. 반환된 dict는 Reducer에 의해 메인 State에 합쳐집니다.

Python · LangGraph 
# Node 1: 의도 파악 함수
def classify_intent(state: SupportState):
    user_msg = state["messages"][-1].content
    # (LLM 호출을 통해 의도를 "refund"로 판단했다고 가정)
    return {"intent": "refund"}  # 수정하고 싶은 필드만 dict로 돌려줌

# Node 2: 주문 정보 조회 함수
def check_order(state: SupportState):
    # state["order_id"]를 읽어서 API 호출 진행
    order_total = 550000  # API 조회 결과 55만 원
    return {"amount": order_total}

# Node 3: 환불 실행 함수
def do_refund(state: SupportState):
    # issue_refund(state["order_id"]) 실행
    return {"messages": [("assistant", "환불이 정상 완료되었습니다.")]}

4-3. Edge — 역과 역을 잇는 철로

노드 간의 실행 순서를 매핑합니다.

  • 정적 엣지 (Static Edge): 무조건 노드 A가 끝나면 B로 실행 흐름을 보냅니다. (g.add_edge("A", "B"))
  • 시작과 끝: 그래프의 시작점을 가리키는 START 엣지와 실행 종착점을 가리키는 END 엣지가 존재합니다.

4-4. compile() — 뼈대 조립하기

정의된 State, Node, Edge를 모아 그래프 인스턴스를 조립하고, 실행 가능한 Runnable 객체로 빌드합니다.

Python · LangGraph 
from langgraph.graph import StateGraph, START, END

# 1. StateGraph에 스키마 등록
g = StateGraph(SupportState)

# 2. 노드 등록
g.add_node("classify", classify_intent)
g.add_node("check_order", check_order)
g.add_node("refund", do_refund)

# 3. 엣지 연결 (정적 흐름)
g.add_edge(START, "classify")
g.add_edge("classify", "check_order")
g.add_edge("check_order", "refund")
g.add_edge("refund", END)

# 4. 컴파일하여 실행용 앱 획득
app = g.compile()
🧭 Section 4 한 줄 요약

LangGraph 구현은 State 스키마를 선언하고, 그 State의 일부 갱신값(dict)을 반환하는 Node 함수들을 작성한 뒤, 그 사이의 길(Edge)을 이어 컴파일(compile)하는 4단계의 명시적 반복이다.

Section 5

5Routing과 제어 흐름 — 분기, 루프, 그리고 사람

선형으로 이어진 파이프라인은 단조롭습니다. LangGraph가 강력한 이유는 조건에 따라 다음 실행 역이 동적으로 갈라지는 분기(Routing)와, 이전 단계로 되돌아가는 순환(Loop), 그리고 사람이 개입해 승인할 때까지 프로세스를 재우는 일시정지(Interrupt)를 가장 완벽하게 지원하기 때문입니다.

5-1. 조건부 엣지 (Conditional Edge) — 라우팅 함수 작성

상태 차트(State)의 현재 값들을 읽어 들여, 다음에 진입해야 할 노드의 이름(문자열)을 반환하는 순수 파이썬 라우팅 지휘기입니다.

Python · LangGraph 
# 의도를 보고 다음 분기를 결정하는 라우팅 함수
def route_intent(state: SupportState) -> str:
    if state["intent"] == "refund":
        return "check_order"
    elif state["intent"] == "track":
        return "track_flow"
    return "general_qna"

# 그래프에 조건부 엣지 등록
g.add_conditional_edges(
    "classify",        # 이 노드가 종료된 후에
    route_intent,      # 이 라우팅 함수를 돌려서
    {                  # 리턴된 문자열에 따라 지정된 노드로 분기
        "check_order": "check_order",
        "track_flow": "track_flow",
        "general_qna": "general_qna"
    }
)

5-2. 루프 (Loop) — Agent의 핵심 심장 박동

LLM이 도구 결과를 받고 계속 돌 것인지 판단하는 ReAct(Reasoning + Acting) 루프를 LangGraph는 엣지의 꼬리물기 순환 구조로 표현합니다.

그림 2. LangGraph에서 에이전트 순환(ReAct Loop)을 노드와 엣지의 순환 구조로 표명한 방식

5-3. 사람 개입 (Human-in-the-Loop) — interrupt & resume

자동 환불 금액이 한도를 넘었을 때, 시스템을 일시정지하고 관리자의 검토 승인을 기다리는 흐름입니다. 바닐라 파이썬에서 이 중단점을 유지하는 건 매우 고통스러운 일이지만, LangGraph 컴파일러는 이를 기본 사양으로 지원합니다.

Python · LangGraph 
# 50만 원이 넘으면 사람의 확인을 받도록 일시 중단하는 노드
def human_approval_node(state: SupportState):
    # interrupt가 실행되면 그래프 실행 엔진이 이 시점에서 물리적으로 '정지'합니다.
    # state 정보를 상담원에게 띄우고 상태가 잠자기 모드(Suspend)로 들어갑니다.
    decision = interrupt({
        "question": "50만 원 초과 환불입니다. 승인하시겠습니까?",
        "amount": state["amount"]
    })
    return {"approved": decision == "APPROVED"}

# 그래프 조립 시 중단점 컴파일 지정
app = g.compile(
    checkpointer=memory_saver,  # 상태 저장을 위한 메모리 saver (Section 6에서 해설)
    interrupt_before=["human_approval_node"]  # 이 노드가 돌기 직전에 실행을 멈추고 제어권을 바깥으로 토스
)

# 1. 봇이 돌다가 멈춤
# thread_config를 통해 고유 대화 키 매핑
config = {"configurable": {"thread_id": "cust-999"}}
app.invoke({"messages": [("user", "55만 원 환불해줘")]}, config=config)
# -> human_approval_node 진입 직전 멈추고 실행 정지됨

# 2. (며칠 뒤 관리자가 승인을 눌렀을 때) 그 자리에서 그대로 이어서 시작 (Resume)
from langgraph.types import Command
app.invoke(Command(resume="APPROVED"), config=config)

LangGraph는 조건부 엣지(Conditional Edge)로 분기를 뚫고, 엣지를 자기 노드로 회항시켜 루프를 그리며, interrupt와 resume으로 사람 승인 흐름을 명시적으로 설계도에 이식할 수 있다.

Section 6

6State와 영속성 — Memory의 토대

에이전트에게 기억력이 없다면 매번 첫 만남처럼 멍청하게 행동할 것입니다. LangGraph는 단기/장기 기억을 구현하기 위해 상태를 디스크나 DB에 꼬박꼬박 기록하는 영속성(Persistence) 메커니즘을 사용합니다.

6-1. Checkpointer — 자동 세이브포인트

LangGraph 그래프를 컴파일할 때 checkpointer를 인자로 심어주면, 그래프가 노드를 하나 통과할 때마다 State의 스냅샷을 자동으로 DB에 직렬화해 저장합니다.

Python · LangGraph 
from langgraph.checkpoint.memory import MemorySaver  # 테스트용 인메모리 저장소
# from langgraph.checkpoint.postgres import PostgresSaver # 실무용 DB 저장소

memory_saver = MemorySaver()
app = g.compile(checkpointer=memory_saver)
💡 비유 — 어드벤처 게임의 자동 세이브

체력을 깎이고 포션을 획득한 주인공이 보스 방 바로 앞 세이브 포인트에 도달하면 게임이 진행 상황을 자동 저장합니다. 게임기가 갑자기 꺼지거나 보스에게 패배해도(서버 종료), 처음부터 다시 깨는 게 아니라 세이브 데이터(Checkpointer)를 읽어 바로 보스 방 앞에서 재개합니다. LangGraph의 영속성이 정확히 이와 같습니다.

이 Checkpointer 하나가 다음과 같은 다채로운 고급 프로덕션 기능을 열어줍니다.

  • 단기 기억 (Session Memory): thread_id별로 대화 상태를 물고 다닙니다.
  • 실패 방어 (Durability): 실행 중간에 에러가 터져 멈춰도 에러난 지점부터 재시도가 가능합니다.
  • 시간 여행 (Time-travel): 과거 특정 노드 상태로 강제로 되감아 "여기서 다른 도구를 썼더라면 어땠을까?" 가상 디버깅이 가능합니다.

6-2. thread_id — 단기 기억의 단위

대화 세션을 구별하는 고유 키값입니다. 같은 thread_id로 봇을 호출하면, 봇은 직전 턴에 쌓아놓은 State(대화 이력 등)를 DB에서 통째로 긁어와 이어서 생각하기 시작합니다.

Python · LangGraph 
config = {"configurable": {"thread_id": "user-chulsoo-session-1"}}

# 첫 번째 대화
app.invoke({"messages": [("user", "안녕, 나는 철수야. 내 주문번호는 1234야.")]}, config=config)
# -> 봇: "반갑습니다 철수님!" (State에 이름과 주문번호 저장됨)

# 두 번째 대화 (같은 thread_id!)
app.invoke({"messages": [("user", "방금 말한 그 주문 상태 알려줘.")]}, config=config)
# -> 봇: "철수님의 주문 1234번은 현재 배송 중입니다." (이전 대화를 완벽히 기억 - 단기 기억)

6-3. 장기 기억 (Long-term Memory) — thread를 넘어서는 상설 수첩

thread_id는 세션이 만료되거나 브라우저 창을 끄면 사라지는 단기 맥락입니다. 반면 "철수는 VIP 고객이다", "환불 시 꼼꼼히 체크받는 걸 싫어한다" 같은 세션을 넘어 영구적으로 기억해야 하는 선호도는 별도의 글로벌 메모리인 Store에 기록합니다.

Python · LangGraph 
# 💻 의사코드 (구조/원리) — thread를 넘는 글로벌 저장공간
# 1. 봇이 대화 중 철수의 선호를 판단해 Store에 기록
store.put(
    namespace=("users", "chulsoo"), 
    key="preferences", 
    value={"strictness": "low", "vip_level": "gold"}
)

# 2. 한 달 뒤, 전혀 다른 세션(새 thread_id)으로 철수가 찾아왔을 때
new_config = {"configurable": {"thread_id": "chulsoo-new-chat-99"}}
# 첫 노드에서 Store를 열어 철수의 성향을 읽고 프롬프트에 주입합니다.
user_pref = store.get(("users", "chulsoo"), "preferences")

📌 LangGraph에서 무엇을 장기 기억으로 남겨 저장할지는 개발자가 노드 안의 코드로 100% 명시적으로 조작하고 저장해야 합니다. (Agno는 이를 LLM이 알아서 훔쳐 듣고 받아 적게 자동화해 줍니다.)

🧭 Section 6 한 줄 요약

LangGraph의 기억력은 노드마다 상태를 세이브하는 Checkpointer(단기 기억)와 세션을 초월해 사용자의 성향을 수동 저장하는 Store(장기 기억)라는 명시적 데이터베이스 조작으로 구현된다.

Section 7

7Workflow와 Agent를 그래프로

우리는 앞서 Section 2에서 Workflow(결정론적 규칙)와 Agent(LLM의 자율 계획)의 차이를 배웠습니다. LangGraph에서는 이 두 가지 형태를 코드로 어떻게 다르게 그리는지 비교해 봅니다.

1. 엄격한 흐름 (Workflow)

프로세스의 실행 궤도를 사람이 짠 코드가 철저하게 보장합니다. LLM은 단순 가공이나 번역, 최종 결과 포맷팅에만 사용됩니다.

Python · LangGraph 
# 🧹 Code Review Workflow 예시
# flow: check_syntax ➔ run_linter ➔ llm_review ➔ END
g = StateGraph(CodeState)
g.add_node("syntax", check_syntax)
g.add_node("linter", run_linter)
g.add_node("llm", ask_llm_review)

# 엣지 연결이 완전 일직선 (결정론)
g.add_edge(START, "syntax")
g.add_edge("syntax", "linter")
g.add_edge("linter", "llm")
g.add_edge("llm", END)
app = g.compile()

2. 자율적인 흐름 (Agent)

LLM이 상태를 보고 스스로 판단해 다음 역을 루프로 골라 잡습니다. 어디로 튀어 나갈지 런타임 이전에는 확답할 수 없습니다.

Python · LangGraph 
# 🔍 Research Agent (ReAct) 예시
# agent 노드가 tools 노드를 무한 순환할 수 있음
g = StateGraph(AgentState)
g.add_node("agent", call_llm_planner)
g.add_node("tools", run_searches)

g.add_edge(START, "agent")
# LLM의 판단에 따라 루프 돌지 정지할지 라우팅
g.add_conditional_edges(
    "agent",
    should_continue,
    {"tools": "tools", "end": END}
)
g.add_edge("tools", "agent")  # 루프 복귀 선로
app = g.compile()

🧪 support-bot 실무에서는 이 둘을 합친 하이브리드형이 대세입니다. "고객의 의도를 읽어 환불 자격 여부를 체크하고, RAG로 정책 조율하는 데까지"는 한 치의 오차가 없어야 하므로 엄격한 Workflow로 돌립니다. 자격이 탈락한 후 "고객에게 정책의 당위성을 설명하고 마음을 달래는 챗봇 응대" 구간은 LLM이 임기응변을 부리도록 자율 Agent로 돌려 엮는 것이 모범 설계입니다.

🧭 Section 7 한 줄 요약

LangGraph는 일직선 엣지만 이은 정적 Workflow와 조건부 엣지를 포함해 순환시킨 자율 Agent를 동일한 그래프 표현 기법으로 자유자재로 믹스하여 표현할 수 있는 아키텍처적 유연성을 보장한다.

Section 8

8LangGraph Multi-Agent & Orchestration

프로젝트의 복잡도가 한계치를 돌파하면, 하나의 거대 그래프는 더 이상 유지보수할 수 없습니다. 이때 여러 명의 소형 전문가 그래프들을 배치하고 조율하는 Multi-Agent(다중 에이전트) 아키텍처가 등판합니다.

8-1. Supervisor Pattern — 감독관과 세 전문가

가장 널리 쓰이는 패턴입니다. 고객의 문의가 들어오면 대장 노드(Supervisor)가 내용을 듣고 어떤 전문가 에이전트(서브그래프)에게 일을 맡길지 결정합니다. 일을 마친 전문가는 리포트를 들고 다시 대장에게 돌아오며(복귀), 대장은 추가 업무를 시키거나 작업을 종료(FINISH)합니다.

그림 3. Supervisor 에이전트가 사용자 질의에 맞춰 각 전문 에이전트에게 업무를 위임하고 결과를 회수하는 허브 앤 스포크 조율 모델

8-2. Subgraph — 복잡성 격리

위의 전문가 에이전트들(환불 전담, 추천 전담) 각각이 사실 단순 노드가 아니라, 자기만의 State와 Node, Edge를 가진 독립된 또 하나의 완결된 LangGraph 그래프입니다. 이 자식 그래프를 부모 그래프의 하나의 노드로 삽입(SubGraph)하여 모듈식 컴포넌트화를 달성합니다.

8-3. 병렬 실행과 Send — 동적 Map-Reduce

리서치 에이전트가 "동시에 5개 해외 사이트를 분할 검색"하거나, support-bot이 "환불 적격성 검사와 블랙리스트 조회를 동시에 진행"하는 경우입니다. LangGraph는 `Send` 명령을 사용해 런타임에 동적으로 N개의 병렬 노드(Worker)를 스폰하고, 실행 결과를 Reducer가 안전하게 한 바구니로 취합하는 구조를 취합니다.

Python · LangGraph 
from langgraph.types import Send

# 1. 런타임에 검색어가 N개 추출되면, search_node를 N개 병렬로 동시에 스폰
def fan_out(state: SupportState):
    queries = ["환불 규정 조회", "결제 수단 대조", "고객 등급 혜택"]
    return [Send("search_node", {"query": q}) for q in queries]

# 2. 병렬 실행 결과는 State의 Reducer가 충돌 없이 차곡차곡 리스트에 추가(Merge)해 줍니다.
# Annotated[list, operator.add] <- 이 리듀서 덕에 동시성 데이터 갱신도 안전합니다.
🧭 Section 8 한 줄 요약

LangGraph의 Multi-Agent 오케스트레이션은 "에이전트 = 서브그래프", "조율 = 부모 그래프"라는 단 하나의 철저한 그래프 멘탈 모델로 수렴되며, 병렬 처리는 Send와 Reducer의 데이터 병합으로 달성한다.

Section 9

9설계 철학 — "Agent가 1급 시민"

이제 완전히 반대편에 있는 철학을 만나볼 시간입니다. LangGraph가 "제어 흐름도를 직접 그려라"라고 지휘봉을 쥐여주었다면, Agno는 "흐름도는 잊고, 필요한 기능(기억·지식·도구)을 갖춘 Agent 객체 하나를 선언적으로 정의하라"고 제안합니다.

9-1. 무엇에 반발해서 태어났나

LangGraph를 써서 간단한 리서치 에이전트 하나를 구현하려고 하면 다음과 같은 생각이 스칩니다. "겨우 날씨 정보 조회해서 보고서 뱉는 에이전트 하나 만드는데 State 스키마 설계하고, 노드 함수 5개 쪼개고, 엣지 잇고, reducer 충돌을 고민해야 한다고? 너무 과도한 보일러플레이트 코드가 아닌가?"

Agno(구 Phidata)는 바로 이 불만에서 탄생했습니다. Agno의 창업자들은 다음과 같이 생각했습니다.

💡 "결국 현업에서 쓰는 에이전트의 90%는 똑같은 재료들의 조합이다. [뇌(LLM) + 손발(Tools) + 지침(Instructions) + 기억(Memory) + 지식(Knowledge)]. 그렇다면 흐름을 매번 그리지 않고, 이 인자들을 싹 모아서 받아서 조율하는 '잘 설계된 Agent 단일 객체' 하나만 주면 충분하지 않나?"

Agno에서 Agent는 더 이상 노드와 엣지의 결합이 아닙니다. 단 하나의 독립적이고 막강한 파이썬 객체입니다. 개발자는 캔버스에 흐름도를 그리는 게 아니라, 이 객체의 옵션(인자)을 채워 넣음으로써 개발을 완료합니다.

9-2. Agno가 내세우는 네 가지 기둥

기둥 의미 개발자 체감 가치
생산성 (Productivity) 단 몇 줄의 파이썬 코드로 동작하는 완결된 에이전트를 조립해 즉시 띄울 수 있습니다. 아이디어 구상부터 데모 시연까지 10분 내 도달.
성능 (Performance) 에이전트 인스턴스의 크기가 아주 작고, 초기화 로드가 거의 없습니다. 수천 개의 개별 에이전트를 동시 구동하는 대규모 다중 에이전트 서비스에 유리.
배터리 포함 (Batteries-included) 메모리(Memory), RAG 지식(Knowledge), 세션 영속화(Storage), 단계적 추론(Reasoning)이 라이브러리 내에 풀 패키지로 기본 탑재되어 있습니다. 타 툴킷과의 조립 번거로움이 없음.
모델 비종속 (Model-agnostic) OpenAI, Anthropic, Gemini, local Ollama 등 20여 개 LLM 공급자를 동일 인터페이스로 조작합니다. 모델 교체 시 단 한 줄의 코드 수정만 필요.
💡 비유 — 수제 부품 차량 조립 vs 완성차 옵션 선택

LangGraph가 엔진, 피스톤, 서스펜션, 전선을 가져와 용접하며 차량을 바닥부터 커스텀 조립하는 것이라면, Agno는 대리점에 가서 카탈로그를 펼쳐 완성차에 원하는 옵션(네비게이션, 선루프, 4WD)을 체크하여 즉시 끌고 나오는 것입니다.

옵션을 체크하는 것만으로 차는 문제없이 잘 굴러갑니다. 다만 차량 자체의 섀시 구조나 배선을 기상천외한 방식으로 개조하려는 고도의 요구사항을 만난다면 옵션 카탈로그만으로는 해결하기 어렵습니다.

🧭 Section 9 한 줄 요약

Agno는 "동사(흐름 표현)"에 집중하는 LangGraph와 달리, 에이전트의 완성체 구성을 최우선시하여 "명사(Agent 능력 선언)"에 방점을 찍은 고수준 생산성 중심 프레임워크다.

Section 10

10Agent 중심 설계 — 하나의 객체에 모든 것

Agno로 프로그램을 작성한다는 것은, Agent라는 클래스의 생성자 인자를 조립해 완성하는 일입니다. LangGraph에서 그렸던 루프와 라우팅이 이 단일 객체 내부로 완전히 봉인되어 감춰집니다.

10-1. Agent = Model + Tools + Instructions (+ 그 이상)

Agno 에이전트 선언의 기본적인 뼈대 구조를 살펴봅니다.

Python · Agno 
from agno.agent import Agent
from agno.models.openai import OpenAIChat

support_agent = Agent(
    model=OpenAIChat(id="gpt-4o"),
    # 손발이 될 도구 함수 리스트
    tools=[track_order, issue_refund, search_policy],
    # 셰프가 지켜야 할 업무 규칙 자연어 지침
    instructions=[
        "너는 SaaS support-bot 상담원이다.",
        "환불은 무조건 check_policy 도구를 사용해 적격성을 검사한 뒤 실행해야 한다."
    ],
    markdown=True  # 마크다운 포맷팅 출력 강제
)

# 실행 및 출력
support_agent.print_response("주문 1234번 환불해줘")

📌 이 단순성을 LangGraph의 복잡성과 대조해 보십시오.
LangGraph에서 엣지와 라우팅으로 명시했던 ReAct 순환 루프(LLM 판단 ➔ 도구 호출 ➔ 결과 수집 ➔ 재판단)가 Agent의 print_response() 혹은 run() 함수 내부에서 암묵적으로 알아서 실행됩니다.

10-2. Instructions — 흐름 대신 자연어 지침으로 훈육하기

Agno에서는 흐름도를 강제하지 않는 대신, instructions 배열에 자연어로 가이드라인을 적어 LLM이 스스로 올바른 단계를 밟도록 유도합니다.

⚠️ 함정 — 자연어 지침은 강제가 아닌 권장 사항이다

LLM은 똑똑하지만 자연어 지침을 가끔 건너뛰거나 어길 수 있습니다. 예를 들어 "50만 원 넘는 환불은 절대 하지 말라"고 프롬프트를 줬지만, 고객의 화난 어조에 LLM이 당황해 그냥 환불 승인 결정을 도구에 흘릴 수 있습니다.

따라서 돈이나 개인정보 등 절대 타협할 수 없는 하드 제약(Hard Constraints)은 instructions에만 의존하지 말고, 도구(파이썬 함수) 내부의 첫 줄에 코드로 검증 장치를 박아두거나(Section 11), Agno의 Workflow 모드(Section 13)를 꺼내야 합니다.

11-2. Agentic RAG — 스스로 도구를 쓸지 말지 결정하는 RAG

전통적인 RAG 파이프라인은 고객의 입력이 들어오면 항상, 무조건 벡터 DB로 달려가 문서를 긁어옵니다. 반면 에이전트 세계의 **Agentic RAG(에이전트적 RAG)**는 전혀 다릅니다.

📖 전통적인 RAG (Static Pipeline)

"안녕?" ➔ [검색] ➔ 관련 문서(없음) ➔ LLM ➔ "안녕하세요!"

인사나 잡담 등 지식 검색이 전혀 필요 없는 경우에도 무조건 벡터 DB 조회를 때려 서버 리소스와 API 속도를 지연시킵니다.

🧠 Agentic RAG (Agno 방식)

"안녕?" ➔ LLM 판단: 지식 필요 없음 ➔ 즉시 "안녕하세요!"

"전자제품 환불 정책은?" ➔ LLM 판단: 지식 필요함 ➔ 지식 도구 검색 ➔ 결과 수렴 ➔ 답변

검색 행위가 에이전트가 선택적으로 사용할 수 있는 '하나의 편리한 도구'로 흡수된 형태입니다.

11-3. Knowledge Base 꽂기

Agno에서는 임베딩과 벡터 DB 테이블 매핑을 객체 하나로 묶어 에이전트 생성자에 넘깁니다.

policy_kb.load(recreate=True)

12Memory · Session · Storage in Agno

LangGraph는 모든 대화 기억을 "State를 어떻게 영속화할 것인가"라는 하나의 통합된 문제로 다룹니다. 반면 Agno는 이를 더 명확하고 직관적인 세 가지 특화 기법으로 쪼개어 인자 형태로 제공합니다.

12-1. 기억의 세 가지 층위

개념 역할 LangGraph 대응 실무적 의미 (SaaS 기준)
Session (세션) 현재 진행 중인 단일 대화의 메세지 내역을 보유합니다. (단기 활성 메모리) State 내 messages 배열 현재 브라우저 채팅창에서 나누고 있는 대화 맥락
Storage (저장소) 세션의 내용을 데이터베이스(SQLite, PostgreSQL 등)에 영구 저장합니다. Checkpointer 서버가 재부팅되어도 기존 세션 ID로 대화창을 복구하는 기반
Memory (장기 기억) 세션의 국한을 넘어, 해당 사용자 ID(User ID)의 개인적인 성향이나 정보를 보관합니다. Store 한 달 뒤에 새로운 대화 세션을 파서 찾아와도 사용자의 특성을 알고 있음

12-2. SQLite Storage 적용하기

Agno 에이전트에 영속적 저장 능력을 추가하려면 storage 인자에 DB 커넥션을 꽂아주면 됩니다. 대화 내역이 자동으로 SQLite에 저장되며, session_id를 활용해 이력을 다시 복원합니다.

Python · Agno 
from agno.agent import Agent
from agno.storage.sqlite import SqliteStorage

# SQLite 세션 데이터베이스 세팅
sqlite_db = SqliteStorage(table_name="agent_sessions", db_file="tmp/chats.db")

agent = Agent(
    model=OpenAIChat(id="gpt-4o"),
    tools=[track_order],
    storage=sqlite_db,               # storage 장착
    add_history_to_messages=True,   # LLM 호출 시 과거 메세지 자동 로드 주입
    num_history_runs=5              # 직전 5턴의 대화만 컨텍스트에 얹음 (토큰 절약)
)

# 세션 ID를 지정해 첫 실행
agent.print_response("주문 ORD-100번 조회해줘", session_id="user_chulsoo_chat")
# (프로세스 완전 재부팅 후) 동일 세션 ID로 대화 재개 시 자동 복원
agent.print_response("그 주문 결제수단이 뭐였지?", session_id="user_chulsoo_chat")

12-3. 장기 기억의 마법 — enable_user_memories

Agno의 가장 매력적인 "배터리 포함" 기능 중 하나는 enable_user_memories=True 옵션입니다. 개발자가 기록 명령을 코딩하지 않아도, 에이전트가 대화 중간에 유의미한 정보라고 판단한 것을 스스로 골라내 사용자 메모리에 덤프합니다.

agent = Agent(
    model=OpenAIChat(id="gpt-4o"),
    memory=Memory(),               # 장기 기억 장치 장착

)

# 1. 고객이 슬쩍 정보를 흘림
agent.print_response("나 환불 까다롭게 검사하는 거 진짜 귀찮아해.", user_id="chulsoo")
# -> Agno가 내부적으로 chulsoo의 성향 DB에 "검사 꼼꼼하게 받는 것을 귀찮아함"을 자동 인서트함

# 2. 며칠 뒤 다른 대화방에서chulsoo가 찾아왔을 때
agent.print_response("이거 환불 요청 넣어줘", user_id="chulsoo")
# -> 봇이 그 선호를 미리 DB에서 로딩해 알고 있음
💡 비유 — 능숙하게 포스트잇을 붙이는 비서

LangGraph의 Store는 수동 장부입니다. 사장님(개발자)이 "이 사실 중요하니까 수첩 5페이지에 기록해"라고 정확히 명령 코드를 짜야 적습니다. 반면 Agno의 자동 기억은 센스 넘치는 개인 비서입니다. 사장님이 시키지 않아도 손님의 흘러가는 말투 속 선호도("따뜻한 물만 드심")를 캐치해 고객 카드에 포스트잇으로 슥 붙여놓습니다.

🧭 Section 12 한 줄 요약

Agno는 기억력을 Session(대화) · Storage(세션 DB 영속화) · Memory(장기 선호)로 세분화해 인자 설정으로 켜며, 특히 LLM이 스스로 유용한 유저 사실을 추출·수렴하는 자동화 장기기억을 내장하고 있다.

13-2. Research Team 실제 코드 구현

13-3. Agno Workflow — 결정론의 방어벽

Agno의 기본 에이전트가 자율성(비결정론)에 치우쳐 있다면, 엄격한 실행 제어를 원할 때는 Workflow 클래스를 선속해 파이썬 제어문(if, for)으로 프로세스를 강제할 수 있습니다.

Python · Agno 
from agno.workflow import Workflow, step

# 💻 의사코드 (구조/원리) — Agno의 결정론적 Workflow 구현
class RefundWorkflow(Workflow):
    
    @step()
    def pull_order_info(self) -> dict:
        # 1. 무조건 주문 정보부터 조회하도록 코드로 박음
        return check_order_api(self.input["order_id"])

    @step()
    def check_policy(self, order: dict) -> bool:
        # 2. 다음으로 항상 정책 검사 실행
        return policy_rag_search(order)

    @step()
    def process_refund(self, order: dict, eligible: bool) -> str:
        # 3. 분기 판단을 LLM이 아닌 파이썬 'if'문이 강제함
        if not eligible:
            return "환불 거부 사유 안내"
        if order["price"] > 500000:
            return "상담원 에스컬레이션 승인 접수"  # (사람 개입 분기)
        return execute_refund_api(order)

🛡️ LangGraph 아키텍처
  • State 설계: TypedDict로 상태 데이터 명시적 정의
  • Node 구현: classify, check, refund 함수 작성
  • Edge 연결: 노드 사이의 결로와 조건부 분기를 명시적 코딩
  • Checkpointer: 세션 스냅샷 백업 지점 선언
  • 결과: 코드 60~120줄. 제어 흐름이 그래프 설계도로 투명하게 드러남
⚡ Agno 아키텍처
  • Agent 선언: 단일 Agent() 인스턴스 빌드
  • 인자 조립: model, tools, instructions 대입
  • 배터리 가동: knowledge, memory, storage 인자 활성화

  • 결과: 코드 15~30줄. 복잡한 배관이 감춰져 빠르게 결과 획득

14-3. 관점별 깊이 — 수동 변속기 vs 자동 변속기

두 프레임워크는 우열 관계가 아닙니다. 용도의 다름입니다.

💡 비유 — 스틱(Manual) 자동차 vs 오토(Automatic) 자동차

LangGraph는 스틱 변속기 차량입니다. 왼발로 클러치를 밟고 오른손으로 기어 레버를 당겨야 출발합니다. 시동을 꺼뜨리기 십상이고 조작이 어렵습니다. 하지만 레이싱 트랙(정밀 제어와 최적화가 극도로 중요한 미션 크리티컬 시스템)에서는 엔진 RPM을 원하는 만큼 쥐어짜내는 최고의 도구입니다.

Agno는 자동 변속기 차량입니다. 엑셀만 밟으면 기어가 알아서 들어갑니다. 누구나 쉽게 도심을 부드럽게 주행할 수 있습니다. 90%의 일반적 주행(기본 Q&A 및 간단한 어시스턴트)에서는 자동 기어가 압도적으로 편합니다. 다만 급격한 경사로에서 수동으로 기어비를 강제 조정하고 싶을 때는 자동 기어박스라는 감춰진 마법이 답답하게 느껴질 수 있습니다.

🧭 Section 14 한 줄 요약

LangGraph는 배관 코드가 다 드러나는 대신 완벽한 통제(스틱 변속기)를, Agno는 세부 배관을 숨겨 최상의 생산성(자동 변속기)을 주므로 아키텍처 성격에 맞춰 선택해야 한다.

16-3. 자가 검검 6문항

본 교육 문서를 제대로 마스터했는지 자가 점검해 봅니다.

Q1. 왜 Framework인가?
자율 루프 제어, 사람 개입, 영속 세션 등 프로덕션 배관 코드를 바닥부터 짜는 수고를 덜기 위해서.
Q2. LangGraph 탄생 배경?
초기 선형 체인의 한계를 돌파하고, 블랙박스화된 루프 제어권을 그래프로 꺼내 명시하기 위해.
Q3. Agno 탄생 배경?
매번 흐름도를 그리는 보일러플레이트를 타파하고, Agent 클래스 옵션 조립으로 빠른 생산성을 획득하기 위해.
Q4. State 관리의 차이?
LangGraph는 개발자가 State 자료형과 Reducer를 설계해 지배하고, Agno는 세션과 메모리로 감춰 처리함.
Q5. 에이전트 시대 RAG?
RAG가 전체 시스템이 아니라, 필요할 때만 불러 쓸 수 있는 에이전트의 '하나의 도구'로 수렴함 (Agentic RAG).
Q6. 선택의 핵심 기준?
돈·위험·규제·감사는 LangGraph. MVP·콘텐츠·속도·가벼운 멀티에이전트는 Agno. 실무는 둘의 하이브리드 혼용.

1. 만능 단일 거대 Agent 하나의 에이전트에 도구 40개와 뚱뚱한 프롬프트를 다 줌 ➔ 정신 분열, 도구 오발사 급증 역할이 작고 선명한 3~4개의 에이전트로 쪼개고 Multi-Agent(Team)로 협업 설계
2. 프롬프트 방임주의 "50만 원 넘는 환불은 금지"라는 하드 제약을 instructions 자연어로만 단속 ➔ LLM이 가끔 어기고 송금해버림 절대 지켜야 하는 금전적, 법적 제약은 **도구(파이썬 함수)의 첫머리에 코드로 이중 방어막**을 세워 탈출시킴
3. 평가 지표 없는 출시 눈으로 대강 3번 테스트하고 "답 잘하네" 하고 배포 ➔ 실사용 고객 유입 시 환각 및 엣지 케이스 폭발 가장 빈번한 고객 질문 시나리오 100개 세트를 정답(Ground Truth)과 함께 구축해 두고, 배포 전 **정기 평가 자동화**
4. 무한 도구 호출 방치 에이전트가 답을 못 찾아 동일 도구를 50번 반복 호출 ➔ 한 번의 질문에 몇 만 원의 토큰 비용 덤프 LangGraph의 recursion_limit을 10~15로 낮게 제어하거나, Agno의 max_steps 안전 가드를 제한
5. 자동 장기 기억의 맹신 Agno의 자동 유저 기억에 일임 ➔ 고객의 헛소리나 민감한 프라이버시 정보까지 선호도 DB에 축적됨 장기 기억에 저장되는 정보들의 스키마를 필터링하고 모니터링 관리하는 백오피스 감사 로직 연동
6. 전역 딕셔너리 남발 프레임워크의 State를 쓰지 않고 파이썬 global dict에 모든 상태를 쑤셔 박음 ➔ 비동기 병렬 처리 시 데이터 꼬임 LangGraph의 명시적 State와 Reducer의 규칙 하에 상태 병합을 강제하고 갱신을 추적
7. 한 프레임워크 종교화 "우리 팀은 무조건 LangGraph만 쓴다" ➔ 간단한 Q&A 봇 개발에 3주 소요 및 유지보수 불능 흐름의 성격에 맞춰 도구를 선택. 뼈대는 LangGraph로, Q&A 부품은 Agno로 혼용 조립

부록 C. 핵심 용어 사전

용어 한 줄 설명
Orchestration (오케스트레이션) 여러 LLM 싱킹과 도구, 사람 개입, 분기를 유기적으로 엮어 신뢰성 있는 하나의 시스템으로 굴리는 행위
ReAct Loop (리액트 루프) 생각(Reasoning)과 행동(Acting)을 반복하며 도구 결과를 받아 답변이 완성될 때까지 순환하는 에이전트 코어 메커니즘
Checkpointer (체크포인터) 노드 통과 단계마다 State 전체를 저장 장치에 자동 세이브하여 단기기억 복구 및 에러 복구의 토대를 만드는 장치
Store (스토어) 대화 세션을 초월해 사용자의 성향이나 전역 정보를 상설 기록해 두는 영구 키-값 저장소
Agentic RAG (에이전트적 RAG) 고정 파이프라인에서 탈피해, 에이전트가 답변을 위해 지식이 정말로 필요한 순간에만 선택적으로 수행하는 RAG 구조
Reducer (리듀서) 노드가 반환한 새로운 데이터 조각을 메인 State에 덮어쓸 것인지, 기존 리스트에 결합할 것인지 규정하는 병합 함수
Human-in-the-Loop (HITL) 중요 분기나 과도한 자금 집행 직전에 프로세스를 멈춰 서고 사람이 결정할 때까지 대기했다 이행하는 패턴

부록 D. "이것만은 기억하세요" — 최종 7문장 요약

🧭 핵심 7문장 요약
  1. 진짜 골치 아픈 건 LLM 단독 호출이 아닌 "호출들을 엮는 오케스트레이션 배관"이다.
  2. Workflow(코드가 제어)와 Agent(LLM이 제어)는 트레이드오프 관계이며 실무는 이를 섞어 쓴다.
  3. LangGraph = "Agent를 그래프로". 보일러플레이트가 있는 대신 완벽한 명시적 통제와 시각적 디버깅을 준다.
  4. Agno = "Agent가 1급 시민". 조립형 객체를 제공해 압도적인 개발 속도와 생산성을 보장한다.
  5. 두 진영의 가장 큰 설계상 결별점은 State와 Memory의 갱신 제어권(개발자 직접 조율 vs 프레임워크 자동 대행)이다.
  6. RAG는 에이전트 아키텍처 내에서 독립된 특수 '지식 도구(Knowledge Tool)'의 위상으로 녹아든다.
  7. 금융·승인·규제·감사는 LangGraph, 빠른 MVP·리서치·콘텐츠는 Agno를 고르고, 궁극의 프로덕션은 둘을 믹스한다.
