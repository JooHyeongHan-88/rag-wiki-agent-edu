## Section 1. AI의 기억 문제 — 똑똑한데 왜 자꾸 틀릴까?

RAG를 이해하려면, 먼저 **ChatGPT 같은 LLM이 가진 근본적인 약점**부터 알아야 합니다. RAG는 바로 이 약점을 메우려고 탄생했기 때문입니다.

### 1-1. LLM은 정말 '문서를 기억'하고 있을까?

많은 사람이 ChatGPT를 "엄청나게 많은 문서를 외우고 있는 AI"라고 생각합니다. **절반만 맞습니다.**

LLM(Large Language Model, 거대 언어 모델)은 인터넷의 방대한 글을 학습하면서 "어떤 단어 다음에 어떤 단어가 올 확률이 높은가"를 익힌 **확률 기반 문장 생성기**입니다. 학습 과정에서 지식이 '모델의 신경망 가중치'라는 형태로 **녹아들어가긴** 하지만, 도서관처럼 원문을 그대로 보관하고 찾아보는 것이 아닙니다.

> 💡 **비유 — 시험 전날 벼락치기한 학생**
>
> LLM은 시험 전날 교과서 수백 권을 빠르게 훑고 시험장에 들어간 똑똑한 학생과 같습니다.
> - 전체적인 내용과 흐름은 잘 기억합니다. (→ 일반 상식, 글쓰기, 요약은 잘함)
> - 하지만 "12쪽 셋째 줄에 정확히 뭐라고 쓰여 있었지?"는 헷갈립니다. (→ 구체적 사실은 부정확)
> - 게다가 **교과서를 시험장에 가져갈 수 없습니다.** (→ 우리 회사 문서를 모름)

📌 **핵심:** LLM은 "기억을 떠올려 그럴듯하게 말하는" 데 강하지, "특정 문서를 정확히 인용하는" 데 강한 것이 아닙니다.

### 1-2. Context Window — AI의 '책상 크기'

LLM에게 질문할 때, 모델은 **그 순간 입력창에 들어온 글자만** 볼 수 있습니다. 이렇게 한 번에 볼 수 있는 텍스트의 최대 분량을 **컨텍스트 윈도우(Context Window)** 라고 합니다.

분량은 **토큰(token)** 단위로 셉니다.

> 💡 **토큰이란?** 단어를 잘게 나눈 조각입니다. 영어는 보통 단어 1개가 1~2토큰, 한국어는 글자 1~2개가 약 1토큰 정도입니다. "출산휴가는 며칠인가요?"는 대략 10토큰 안팎입니다. (정확한 숫자는 모델마다 다릅니다.)

> 💡 **비유 — 책상 크기**
>
> 컨텍스트 윈도우는 AI가 한 번에 펼쳐 놓고 볼 수 있는 **책상의 넓이**입니다.
> 책상이 작으면 자료를 몇 장밖에 못 올려 두고, 책상이 크면 두꺼운 보고서도 통째로 올려 둘 수 있습니다.
> 하지만 아무리 큰 책상도 **무한대는 아닙니다.**

최신 모델들의 책상은 꽤 커졌습니다. 대략 12만 토큰(약 300쪽 분량 책 한 권)부터, 큰 모델은 100만 토큰이 넘기도 합니다. 그래서 이런 질문이 자연스럽게 나옵니다.

> "그럼 그냥 우리 회사 문서 전부를 매번 통째로 넣으면 되는 거 아냐? RAG가 왜 필요해?"

좋은 질문입니다. 하지만 그렇게 못 하는 4가지 이유가 있습니다.

1. **용량의 한계:** 회사 문서는 보통 수십만~수백만 쪽입니다. 아무리 큰 책상도 회사 전체 위키, 10년치 규정과 매뉴얼을 다 못 올립니다.
2. **비용:** LLM은 입력 토큰 양에 비례해 과금됩니다. 매 질문마다 책 한 권을 통째로 넣으면 요금이 폭발합니다.
3. **속도:** 입력이 길수록 답변이 느려집니다.
4. **'중간을 흘려듣는' 현상(Lost in the Middle):** 자료를 아주 길게 넣으면, 모델이 **맨 앞과 맨 뒤는 잘 보지만 중간에 묻힌 핵심은 놓치는** 경향이 연구로 알려져 있습니다. 무작정 많이 넣는다고 정답률이 오르지 않습니다.

📌 **핵심:** "다 집어넣기"는 답이 아닙니다. **필요한 부분만 골라서** 넣어야 합니다. → 이것이 바로 RAG의 핵심 아이디어입니다.

### 1-3. Hallucination — AI가 '자신 있게 지어내는' 문제

LLM의 가장 유명한 약점입니다. **Hallucination(환각)** 은 모델이 사실이 아닌 내용을, 마치 사실인 것처럼 **그럴듯하고 자신감 있게** 만들어내는 현상입니다.

왜 이런 일이 생길까요? LLM은 본질적으로 "다음에 올 가장 그럴듯한 말"을 생성하는 기계이기 때문입니다. **'그럴듯함'이 목표이지 '진실'이 목표가 아닙니다.** 모를 때 "모릅니다"라고 하기보다, 빈칸을 가장 자연스러운 말로 메우려는 경향이 있습니다.

> 💡 **비유 — 모른다고 말하기 싫은 신입사원**
>
> 똑똑하지만 자존심 강한 신입사원에게 "작년 3분기 우리 부서 출장 규정이 어떻게 됐죠?"라고 물으면,
> 정확히 모르면서도 "아마… 1박당 8만 원 한도였던 걸로 기억합니다"라고 **그럴듯하게** 답하는 것과 같습니다. 들으면 맞는 것 같지만, 실제 규정은 다를 수 있습니다.

**실생활 예시:** 존재하지 않는 논문이나 판례를 진짜처럼 인용하는 사고가 실제로 여러 번 보고됐습니다. 해외에서는 변호사가 ChatGPT가 만들어 낸 **가짜 판례**를 법원에 제출했다가 제재를 받은 사례도 있었습니다.

📌 **핵심:** 환각은 모델이 "근거 없이 기억에 의존해 답할 때" 가장 심해집니다. → **근거 문서를 함께 주면** 크게 줄어듭니다. (RAG가 하는 일!)

### 1-4. 최신 정보 문제 — AI는 '입사일이 정해진 직원'

LLM은 특정 시점까지의 데이터로 학습됩니다. 이 시점을 **지식 컷오프(knowledge cutoff)** 라고 합니다. 컷오프 이후의 일은 **태어날 때부터 모릅니다.**

> 💡 **비유 — 어느 날 잠든 뒤 깨어나지 않은 백과사전**
>
> LLM은 "○○년 ○월에 인쇄된 백과사전"과 같습니다. 인쇄 이후에 바뀐 법, 새 제품 가격, 어제 발표된 사내 정책은 그 안에 없습니다. 백과사전에게 "오늘 환율"을 묻는 것과 같죠.

게다가 회사 내부 정보는 **컷오프와 무관하게 처음부터 모릅니다.** 우리 회사 인사 규정, 제품 매뉴얼, 어제 올라온 위키 문서는 공개 인터넷에 없으니 학습된 적이 없습니다.

### 🧭 Section 1 한 줄 요약

> LLM은 똑똑하지만 **(1) 우리 회사 문서를 모르고 (2) 책상(컨텍스트)이 유한하며 (3) 모르면 지어내고 (4) 최신·내부 정보에 약하다.**
> RAG는 이 네 가지 문제를 **"답하기 직전에 관련 문서를 찾아서 함께 건네주는"** 방식으로 한 번에 해결합니다.

## Section 2. RAG의 등장 — '오픈북 시험'이라는 발상

Section 1의 문제들을 풀려는 시도는 여러 가지였습니다. 가장 먼저 떠오르는 방법은 "그럼 우리 회사 데이터로 AI를 다시 가르치자(Fine-tuning)"였습니다. 하지만 곧 한계에 부딪혔습니다.

### 2-1. Fine-tuning(미세조정)의 한계

**Fine-tuning이란?** 이미 학습된 LLM에 우리 데이터를 추가로 학습시켜, **모델의 두뇌(가중치) 자체를 조금 바꾸는** 작업입니다.

> 💡 **비유 — 직원을 '재교육'시키기**
>
> Fine-tuning은 신입사원을 몇 달간 연수원에 보내 **머릿속을 다시 세팅**하는 것과 같습니다. 말투나 일하는 방식(예: "항상 정중한 상담원 톤으로 답하라")을 바꾸는 데는 효과적입니다.

하지만 **'지식을 최신으로 유지'하는 용도로는 적합하지 않습니다.** 이유는 다음과 같습니다.

| 한계 | 설명 |
|------|------|
| **느리고 비쌈** | 규정 한 줄 바뀔 때마다 모델을 다시 학습시켜야 합니다. 연수원에 매번 보내는 셈이라 시간·비용이 큽니다. |
| **'외운 것'이 부정확** | 사실을 가중치에 욱여넣어도, 모델은 그것을 **정확히 인용**하기보다 여전히 '그럴듯하게' 떠올립니다. 환각이 사라지지 않습니다. |
| **출처가 없음** | "이 답변은 어느 문서 몇 쪽에서 나왔다"를 댈 수 없습니다. 규정·법률·금융에서는 치명적입니다. |
| **기존 능력 손상 위험** | 새 내용을 무리하게 학습시키면 원래 잘하던 것을 잊는 **'파국적 망각(catastrophic forgetting)'** 이 생길 수 있습니다. |
| **보안·접근권한 처리 어려움** | "이 문서는 임원만 열람 가능" 같은 권한별 차단을 모델 내부에 녹이기 어렵습니다. |

⚠️ **흔한 오해:** "회사 데이터를 Fine-tuning하면 AI가 우리 회사를 다 알게 된다." → 실제로는 **지식 주입에는 비효율적**이고, 자주 바뀌는 정보에는 더더욱 부적합합니다. Fine-tuning의 진짜 강점은 **'무엇을 아는가'가 아니라 '어떻게 행동·표현하는가'를 바꾸는 것**입니다.

### 2-2. 기업 문서 활용 문제

기업이 진짜로 원하는 것은 이것입니다.

> "우리 회사의 **수많은 내부 문서**(규정, 매뉴얼, 계약서, 회의록, 위키)를 근거로,
> AI가 **정확하고, 출처가 분명하며, 항상 최신**인 답을 해 줬으면 좋겠다."

이 요구는 Fine-tuning만으로는 충족되지 않습니다. 문서는 매일 추가·수정되고, 부서별로 접근 권한이 다르며, "근거가 어디냐"를 반드시 댈 수 있어야 하기 때문입니다.

### 2-3. 지식 최신성 문제

세상은 계속 바뀝니다. 법이 개정되고, 가격이 오르고, 어제 새 정책이 공지됩니다. **모델을 통째로 다시 학습시키지 않고도** 지식을 즉시 갱신할 방법이 필요합니다.

### 2-4. 그래서 발상을 바꿨다 — "외우게 하지 말고, 찾아보게 하자"

여기서 RAG의 핵심 통찰이 등장합니다.

> 💡 **RAG = 폐쇄형 시험을 '오픈북 시험'으로 바꾸기**
>
> - **기존 LLM:** 책 없이 머리로만 푸는 **폐쇄형 시험.** → 헷갈리면 지어냄(환각).
> - **Fine-tuning:** 시험 전에 학생을 **재교육**하기. → 비싸고 느리고, 새 내용 반영이 어려움.
> - **RAG:** 시험을 **오픈북**으로 바꾸기! 문제를 받으면 → 관련 페이지를 **즉시 찾아** → 그 페이지를 **보면서** 답을 작성. 📖

오픈북 시험에서는 똑똑한 학생이 **틀릴 이유가 거의 없습니다.** 책에 있는 내용을 보고 답하니까요. 게다가 "몇 쪽을 봤다"고 출처도 댈 수 있고, 책 내용만 바꾸면 답도 즉시 최신이 됩니다.

📌 **RAG의 정의:**
> **RAG(Retrieval-Augmented Generation, 검색 증강 생성)** 란, LLM이 답을 생성하기 **직전에**, 질문과 관련된 문서를 **외부 지식 창고에서 검색(Retrieval)** 해 와서, 그 내용을 **함께 넣어(Augmented)** 답을 만들게(Generation) 하는 방법입니다.

이름을 세 글자로 풀면 그대로 동작 순서가 됩니다.

- **R — Retrieval(검색):** 질문과 관련된 문서를 외부 창고에서 **찾아온다.**
- **A — Augmented(증강):** 찾아온 문서를 질문과 **함께** LLM 입력에 **더한다.**
- **G — Generation(생성):** LLM이 그 근거를 바탕으로 **답을 만든다.**

| 항목 | RAG가 주는 이점 |
|------|----------------|
| 정확성 | 근거 문서를 보고 답하므로 환각이 크게 감소 |
| 출처 | "어느 문서에서 나왔는지" 인용 가능 |
| 최신성 | 모델을 안 건드리고 **문서만 갱신**하면 즉시 반영 |
| 비용 | 통째로 재학습할 필요 없음 |
| 보안 | 사용자 권한에 따라 검색 대상 문서를 제어 가능 |

### 🧭 Section 2 한 줄 요약

> AI를 매번 **재교육(Fine-tuning)** 하는 대신, 답하기 직전에 **관련 문서를 찾아 쥐여주는 '오픈북 시험'(RAG)** 으로 바꾸면 — 정확성·출처·최신성·비용 문제가 한꺼번에 풀린다.

<a id="section-3"></a>
## Section 3. RAG 한눈에 보기 — 전체 흐름도

RAG는 8단계로 이루어집니다. 먼저 전체 그림을 머릿속에 새기고, 다음 단원부터 하나씩 자세히 봅니다.

### 3-1. RAG 전체 흐름 (ASCII Flow)

```
        ┌──────────────────────┐
        │       Document       │   1) 원본 문서 (PDF, Word, 위키 등)
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │       Chunking       │   2) 문서를 의미 단위로 잘게 쪼갬
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │      Embedding       │   3) 각 조각을 '의미 좌표(숫자)'로 변환
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │   Vector Database    │   4) 의미 좌표를 저장·색인 (창고에 정리)
        └──────────┬───────────┘
                   │
══════════════════╪══════════════════  ← 여기까지는 '미리 준비'(오프라인)
                   │                      아래부터는 '질문이 들어올 때'(실시간)
                   ▼
        ┌──────────────────────┐
        │      Retrieval       │   5) 질문과 가장 비슷한 조각을 검색
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │ Prompt Augmentation  │   6) 질문 + 검색결과를 합쳐 프롬프트 구성
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │         LLM          │   7) 근거 자료를 보며 답변 생성
        └──────────┬───────────┘
                   │
                   ▼
        ┌──────────────────────┐
        │        Answer        │   8) 출처가 있는 정확한 답변
        └──────────────────────┘
```

### 3-2. 가장 중요한 구분 — '준비'와 '응답'은 다른 시점이다

> 📌 RAG를 처음 배울 때 가장 많이 헷갈리는 지점입니다. 이것만 잡아도 절반은 이해한 것입니다.

위 흐름은 사실 **두 개의 다른 시점**으로 나뉩니다.

**① 준비 단계 (Indexing / Ingestion) — 미리, 한 번 (오프라인)**
`Document → Chunking → Embedding → Vector DB`
- 마치 **도서관에 새 책이 들어오면 사서가 미리 분류하고 서가에 꽂아 두는** 작업입니다.
- 사용자가 질문하기 **전에** 미리 해 둡니다. 문서가 추가/수정될 때만 다시 합니다.

**② 응답 단계 (Query / Retrieval & Generation) — 질문할 때마다 (실시간)**
`(질문) → Retrieval → Prompt Augmentation → LLM → Answer`
- **이용자가 와서 질문하면**, 사서가 관련 책을 찾아다 주고, 그걸 보고 답을 적어 주는 작업입니다.
- 사용자가 질문할 **때마다** 실행됩니다.

> 💡 **도서관 비유로 전체 묶기**
> - **Chunking** = 책을 통째로 두지 않고 **주제별 색인 카드**로 정리
> - **Embedding** = 각 카드에 **'의미 주소'** 를 부여
> - **Vector DB** = 그 카드들을 **의미 주소순으로 보관하는 특별한 서가**
> - **Retrieval** = 질문을 들고 온 이용자에게 **사서가 관련 카드를 찾아다 줌**
> - **Prompt Augmentation** = 찾은 카드를 질문과 함께 **똑똑한 직원에게 전달**
> - **LLM** = 그 자료를 읽고 **사람이 이해할 문장으로 답을 작성**

### 3-3. 각 단계 한 줄 설명

| # | 단계 | 한 줄 설명 | 비유 |
|---|------|-----------|------|
| 1 | **Document** | 우리가 활용할 원본 자료 | 도서관에 들어온 책 |
| 2 | **Chunking** | 긴 문서를 적당한 크기 조각으로 분할 | 책을 색인 카드로 분리 |
| 3 | **Embedding** | 각 조각의 '의미'를 숫자 벡터로 변환 | 카드에 의미 주소 부여 |
| 4 | **Vector DB** | 벡터들을 저장하고 빠르게 검색 가능하게 색인 | 의미순 정리 서가 |
| 5 | **Retrieval** | 질문과 의미가 가까운 조각 Top-K개 검색 | 사서가 관련 카드 찾기 |
| 6 | **Augmentation** | 검색한 조각을 질문과 합쳐 프롬프트 구성 | 직원에게 자료+질문 전달 |
| 7 | **LLM** | 받은 자료를 근거로 자연스러운 답변 생성 | 직원이 답안 작성 |
| 8 | **Answer** | 출처가 있는 최종 답변 | 이용자에게 답변 전달 |

### 🧭 Section 3 한 줄 요약

> RAG는 **미리 문서를 정리해 두는 '준비 단계(Chunking→Embedding→Vector DB)'** 와, **질문마다 관련 조각을 찾아 답하는 '응답 단계(Retrieval→Augmentation→LLM)'** 로 나뉜다.

<a id="section-4"></a>
## Section 4. Ingestion Pipeline — 문서를 'AI가 찾을 수 있는 형태'로 준비하기

**Ingestion(인제스천)** 은 '섭취·받아들이기'라는 뜻으로, 원본 문서를 RAG가 검색할 수 있는 형태로 **가공해 창고에 넣는 전 과정**을 말합니다. (Section 3의 '준비 단계'를 더 자세히 본 것입니다.)

> 💡 **비유 — 식자재를 손질해 냉장고에 정리하기**
>
> 시장에서 사 온 식자재(원본 문서)를 그대로 냉장고에 던져 넣으면 요리할 때 못 찾습니다.
> 씻고(Cleaning), 손질해 썰고(Chunking), 라벨을 붙여(Embedding), 칸별로 정리(Indexing)해 둬야 필요할 때 바로 꺼내 씁니다.

### Ingestion 6단계 흐름

```
[원본 문서]
   │
   ▼ (1) Document Loading  — 문서를 읽어 들이기
   ▼ (2) Parsing           — 텍스트/표/구조 추출
   ▼ (3) Cleaning          — 잡음 제거·정리
   ▼ (4) Chunking          — 의미 단위로 분할
   ▼ (5) Embedding         — 조각을 벡터로 변환
   ▼ (6) Indexing          — 벡터 DB에 저장·색인
   │
   ▼
[검색 준비 완료된 Vector DB]
```

각 단계를 **목적 / 입력 / 처리 / 출력**으로 정리합니다.

### (1) Document Loading — 문서 불러오기

- **목적:** 여기저기 흩어진 다양한 형식의 문서를 시스템 안으로 읽어 들인다.
- **입력:** PDF, Word(.docx), Excel, PPT, HTML 웹페이지, Notion·Confluence 위키, 이메일, DB 레코드 등
- **처리:** 각 형식에 맞는 '로더(Loader)'로 파일을 열어 내용을 가져온다.
- **출력:** 원본 텍스트 + 부가정보(파일명, 페이지, 작성일 등 **메타데이터**)

> 🔧 실무 팁: 같은 'PDF'라도 글자로 된 PDF와 **스캔(이미지) PDF**는 다릅니다. 스캔본은 글자를 인식하는 **OCR** 과정이 추가로 필요합니다.

### (2) Parsing — 구조 해석하기

- **목적:** 줄글뿐 아니라 **표, 제목, 목록, 단락 구조**를 살려서 추출한다.
- **입력:** 로더가 읽은 원본 문서
- **처리:** 제목과 본문 구분, 표를 행·열 구조로 인식, 머리말/꼬리말 식별 등
- **출력:** 구조 정보가 보존된 텍스트

> ⚠️ 표를 잘못 해석하면 "출산휴가 90일"의 숫자와 항목이 어긋나 엉뚱한 답이 나옵니다. 표·양식이 많은 문서일수록 파싱 품질이 정답률을 좌우합니다.

### (3) Cleaning — 잡음 제거하기

- **목적:** 검색·답변에 방해되는 불필요한 요소를 제거해 품질을 높인다.
- **입력:** 파싱된 텍스트
- **처리:** 반복되는 머리말/꼬리말·페이지 번호 제거, 광고·내비게이션 메뉴 제거, 깨진 문자·중복 공백 정리, (필요 시) 개인정보 마스킹
- **출력:** 깔끔하게 정돈된 텍스트

> 💡 비유: 요리 전에 채소를 **씻고 다듬는** 단계. 흙(잡음)이 묻은 채로 썰면 요리 전체를 망칩니다.

### (4) Chunking — 의미 단위로 쪼개기

- **목적:** 긴 문서를 검색·전달에 알맞은 크기의 조각으로 나눈다. (자세한 내용은 **Section 5**)
- **입력:** 정제된 텍스트
- **처리:** 문단·문장 경계를 고려해 일정 크기(예: 300~500자)로 분할, 조각 간 약간 겹침(overlap)
- **출력:** 여러 개의 **청크(chunk)** + 각 청크의 메타데이터

### (5) Embedding — 의미를 숫자로 바꾸기

- **목적:** 각 청크의 '의미'를 컴퓨터가 비교할 수 있는 숫자 벡터로 변환한다. (자세한 내용은 **Section 6**)
- **입력:** 청크들
- **처리:** 임베딩 모델에 각 청크를 넣어 벡터(숫자 배열)를 얻는다.
- **출력:** `청크 텍스트 + 벡터 + 메타데이터` 묶음

### (6) Indexing — 벡터 DB에 저장·색인

- **목적:** 벡터들을 저장하고, 나중에 **빠르게** 비슷한 것을 찾을 수 있도록 색인을 만든다. (자세한 내용은 **Section 7**)
- **입력:** 청크별 벡터 + 메타데이터
- **처리:** 벡터 DB에 적재하고 검색용 색인(index) 구조 생성
- **출력:** **질문이 오면 즉시 검색 가능한 상태의 Vector DB**

### Ingestion 단계 요약표

| 단계 | 목적 | 입력 | 출력 |
|------|------|------|------|
| 1. Loading | 문서 읽어오기 | 원본 파일들 | 원시 텍스트 + 메타데이터 |
| 2. Parsing | 구조 해석 | 원시 텍스트 | 구조 보존 텍스트 |
| 3. Cleaning | 잡음 제거 | 구조 텍스트 | 정제 텍스트 |
| 4. Chunking | 조각내기 | 정제 텍스트 | 청크들 |
| 5. Embedding | 벡터화 | 청크들 | 청크+벡터 |
| 6. Indexing | 저장·색인 | 청크+벡터 | 검색 가능한 Vector DB |

### 🧭 Section 4 한 줄 요약

> Ingestion은 원본 문서를 **읽고(Load) → 해석하고(Parse) → 닦고(Clean) → 쪼개고(Chunk) → 숫자로 바꿔(Embed) → 창고에 정리(Index)** 하는, RAG의 '재료 손질' 과정이다. 이 단계의 품질이 답변 품질의 8할을 결정한다.

## Section 5. Chunking — 문서를 어떻게 쪼갤 것인가

### 5-1. Chunking이 왜 필요한가?

청크(chunk)는 '덩어리, 조각'이라는 뜻입니다. Chunking은 긴 문서를 작은 조각으로 나누는 일입니다. 왜 통째로 두지 않고 굳이 쪼갤까요? 세 가지 이유가 있습니다.

1. **검색 정확도:** 100쪽짜리 문서 전체를 하나의 '의미 좌표'로 만들면 주제가 뒤섞여 흐릿해집니다. 작게 쪼개야 각 조각이 **하나의 분명한 주제**를 가져 검색이 정확해집니다.
2. **컨텍스트 절약:** LLM 책상(컨텍스트)은 유한합니다. 필요한 **딱 그 문단 몇 개**만 넣어야 효율적입니다.
3. **정밀한 출처:** "이 답은 7쪽 휴가 규정 조항에서 나왔다"처럼 콕 집어 근거를 댈 수 있습니다.

> 💡 **비유 — 책 vs 색인 카드**
>
> 도서관에서 "출산휴가" 한 줄을 찾는데 1,000쪽 책 전체를 건네받으면 곤란합니다.
> "출산휴가 = 90일"이라고 적힌 **색인 카드 한 장**을 받는 게 훨씬 유용하죠. Chunking은 책을 색인 카드 더미로 바꾸는 일입니다.

> ⚠️ **너무 잘게 쪼개도, 너무 크게 쪼개도 문제입니다.**
> - 너무 작게(예: 한 문장): 문맥이 끊겨 의미가 사라짐. ("90일" → 무엇이 90일인지 모름)
> - 너무 크게(예: 한 챕터): 여러 주제가 섞여 검색이 부정확하고 컨텍스트 낭비.
> - 그래서 **'적당한 크기로 의미를 보존하며' 쪼개는 전략**이 중요합니다.

### 5-2. 네 가지 대표 Chunking 방식

#### ① Fixed Chunking (고정 크기 분할)

- **동작 원리:** 의미를 따지지 않고 **무조건 N글자(또는 N토큰)마다 싹둑** 자른다. (예: 500자마다)
- **장점:** 가장 단순하고 빠르며 구현이 쉽다. 결과가 예측 가능하다.
- **단점:** 문장·단어 중간을 잘라버려 **"출산휴가는 90 / 일입니다" 처럼 의미가 두 동강** 날 수 있다.
- **사용 사례:** 빠른 프로토타입, 형식이 일정한 단순 텍스트.

```
원문:  ......출산전후휴가는 출산 전후로 총 90일이 주어집니다. 다태아의 경우......
              └────────── 500자 ──────────┘└────────── 다음 500자 ──────────┘
                                          ↑ 의미와 무관하게 여기서 싹둑!
```

#### ② Recursive Chunking (재귀적 분할) ⭐ 가장 무난한 기본값

- **동작 원리:** **자연스러운 경계부터** 차례로 시도한다. 먼저 '문단(빈 줄)' 단위로 나누고, 그래도 너무 크면 '문장' → '단어' 순으로 점점 잘게 나눈다.
- **장점:** 문단·문장 구조를 최대한 **보존**해 의미가 덜 깨진다. 대부분 상황에서 무난.
- **단점:** 고정 분할보다 약간 복잡. 의미를 '완벽히' 따지지는 않는다.
- **사용 사례:** **대부분의 실무 RAG의 1순위 기본 선택.** (LangChain의 기본 분할기가 이 방식)

```
1순위: 문단으로 자르기 (\n\n)  ──► 조각이 아직 크면
2순위: 문장으로 자르기 (. ! ?) ──► 그래도 크면
3순위: 단어/글자로 자르기       ──► 목표 크기 이하가 될 때까지
```

#### ③ Semantic Chunking (의미 기반 분할)

- **동작 원리:** 문장들의 **의미(임베딩)** 를 비교해, 주제가 비슷한 문장끼리 묶고 **주제가 바뀌는 지점에서** 자른다.
- **장점:** 한 조각이 하나의 주제로 깔끔하게 묶여 **검색 품질이 가장 좋은 편**이다.
- **단점:** 자를 때 임베딩 계산이 들어가 **느리고 비용이 더 든다.** 구현이 복잡.
- **사용 사례:** 주제가 자주 바뀌는 복잡한 문서, 품질이 비용보다 중요한 경우.

```
문장1 ─┐
문장2 ─┤ (의미 비슷 → 한 묶음)   ◄── Chunk A (주제: 출산휴가)
문장3 ─┘
─────────  ⟵ 의미가 확 바뀜! 여기서 자르기
문장4 ─┐
문장5 ─┘ (새 주제)              ◄── Chunk B (주제: 연차휴가)
```

#### ④ Sliding Window (슬라이딩 윈도우 / 겹침 분할)

- **동작 원리:** 조각을 자를 때 **앞 조각의 끝부분을 다음 조각의 앞에 겹쳐서(overlap)** 넣는다. 창문을 조금씩 밀며 사진을 찍는 것과 비슷.
- **장점:** 경계에 걸친 문장이 잘려도, **겹친 부분 덕분에 문맥이 보존**된다.
- **단점:** 내용이 중복 저장되어 **용량이 늘고 비용이 증가**한다.
- **사용 사례:** 문맥 연결이 중요한 문서. **단독 방식이라기보다, 위 ①~③과 함께 쓰는 보완 옵션**에 가깝다.

```
Chunk1:  [ A B C D E ]
Chunk2:        [ D E F G H ]      ← D, E를 겹쳐서 문맥 유지
Chunk3:              [ G H I J K ]
                  └겹침┘
```

### 5-3. Chunking 방식 비교표

| 방식 | 동작 원리 | 장점 | 단점 | 사용 사례 |
|------|----------|------|------|----------|
| **Fixed** | 일정 글자/토큰마다 기계적으로 자름 | 단순·빠름·예측 가능 | 문장·의미 단절 | 빠른 프로토타입 |
| **Recursive** ⭐ | 문단→문장→단어 순으로 자연 경계 우선 | 구조 보존, 균형 good | 의미를 완벽히 보진 않음 | **대부분의 실무 기본값** |
| **Semantic** | 의미가 바뀌는 지점에서 자름 | 검색 품질 최상 | 느리고 비쌈, 복잡 | 복잡·고품질 요구 문서 |
| **Sliding Window** | 조각을 겹쳐서 자름 | 경계 문맥 보존 | 중복으로 용량↑ | 문맥 연결 중요 (보완용) |

> 🔧 **실무 권장 출발점 (2025 기준 통념)**
> - **일단 Recursive Chunking으로 시작**하세요. 가장 안전한 기본값입니다.
> - 크기는 **약 300~500자(또는 256~512토큰)**, **겹침은 10~20%**(예: 50~100자) 정도가 무난한 출발점.
> - 정답률이 부족하면 그때 Semantic 등으로 **점진적으로 고도화**하세요. 처음부터 복잡하게 갈 필요 없습니다.

### 🧭 Section 5 한 줄 요약

> Chunking은 문서를 검색하기 좋은 '색인 카드' 크기로 쪼개는 일이며, **너무 크지도 작지도 않게, 의미를 보존하며** 쪼개는 것이 핵심이다. 고민되면 **Recursive + 약간의 겹침**으로 시작하라.

## Section 6. Embedding — 의미를 '숫자 좌표'로 바꾸기

이 단원은 RAG에서 가장 중요하면서도, 비개발자가 가장 신기해하는 부분입니다. 천천히 갑니다.

### 6-1. 왜 단어를 '숫자'로 바꿔야 할까?

컴퓨터는 글자를 **의미로** 이해하지 못합니다. 컴퓨터에게 "고양이"와 "야옹이"는 그냥 서로 다른 글자 덩어리일 뿐, 둘이 비슷하다는 걸 모릅니다. 컴퓨터가 잘 다루는 유일한 것은 **숫자**입니다.

그래서 **"단어·문장의 '의미'를 숫자(좌표)로 표현"** 하기로 합니다. 이렇게 만든 숫자 묶음을 **임베딩(Embedding) 벡터**라고 합니다.

> 📌 **벡터(vector)** 란? 그냥 **숫자들을 순서대로 나열한 목록**입니다. 예: `[0.21, -0.88, 0.05, ...]`
> 실제 임베딩은 보통 숫자가 **수백~수천 개**입니다. (예: OpenAI의 한 모델은 1,536개)

핵심 마법은 이것입니다.

> 📌 **임베딩의 핵심 원리:**
> **'의미가 비슷한 것'은 '숫자 좌표도 가깝게'** 만들어진다.

### 6-2. 벡터 공간 — '의미의 지도'

숫자 좌표가 있으면 **위치**를 찍을 수 있습니다. 임베딩은 모든 단어·문장을 거대한 **'의미의 지도(벡터 공간)'** 위의 한 점으로 표현합니다.

> 💡 **비유 — 지도 위의 도시**
>
> 지도에서 '서울'과 '인천'은 가깝고, '서울'과 '부산'은 멉니다. 위도·경도라는 **숫자 두 개**로 도시 위치를 표현했기 때문이죠.
> 임베딩도 똑같습니다. 단, 위도·경도 2개가 아니라 **수백~수천 개의 숫자**로 '의미의 위치'를 표현합니다. 그리고 거리는 물리적 거리가 아니라 **'의미의 가까움'** 입니다.

지도 위에서는 이런 일이 일어납니다.
- 의미가 비슷한 것들끼리 **무리(군집)** 를 이룹니다. (동물끼리, 음식끼리, 교통수단끼리)
- "가깝다 = 의미가 비슷하다"가 성립합니다.

### 6-3. 직접 보기 — "고양이 / 강아지 / 자동차"

설명을 위해 차원을 **딱 2개**로 줄여(실제론 수백~수천 개) 가상의 좌표를 그려 보겠습니다.

| 단어 | (가상) 좌표 [동물성, 생명성] |
|------|------------------------------|
| 🐱 고양이 | `[0.91, 0.88]` |
| 🐶 강아지 | `[0.89, 0.90]` |
| 🚗 자동차 | `[0.10, 0.05]` |

이걸 지도에 찍으면 이렇게 됩니다.

```
 생명성 ▲
   1.0 │            🐱고양이
       │            🐶강아지     ← 둘이 딱 붙어 있음 (의미가 비슷!)
       │
       │
       │
   0.0 │   🚗자동차                ← 멀리 떨어져 있음 (의미가 다름!)
       └─────────────────────────►
        0.0                   1.0  동물성
```

- **고양이 ↔ 강아지:** 좌표가 거의 붙어 있습니다. 둘 다 '털 달린 반려동물'이라 의미가 비슷하니까요.
- **고양이 ↔ 자동차:** 좌표가 멀리 떨어져 있습니다. '동물'과 '기계'는 의미가 전혀 다르니까요.

> 📌 바로 이것이 RAG가 작동하는 비결입니다.
> 사용자가 "반려묘"라고 검색해도, 글자가 다른 "고양이" 문서를 찾아낼 수 있습니다. **글자가 아니라 '의미의 위치'로 찾기 때문**입니다. 이것을 **의미 기반 검색(semantic search)** 이라고 합니다.

### 6-4. 의미적 유사도 — '키워드 검색'과의 결정적 차이

기존 검색(예: Ctrl+F, 옛날 검색엔진)은 **글자가 똑같이 일치**해야 찾습니다.

| 질문 | 키워드 검색 (글자 일치) | 의미 검색 (임베딩) |
|------|------------------------|--------------------|
| "출산휴가 며칠?" → 문서엔 "출산전후휴가 기간" | ❌ 글자가 달라 못 찾음 | ✅ 의미가 같아 찾아냄 |
| "노트북 안 켜져요" → 문서엔 "전원이 안 들어올 때" | ❌ 못 찾음 | ✅ 찾아냄 |
| "퇴사 절차" → 문서엔 "이직 시 인수인계" | ❌ 못 찾음 | ✅ 관련 문서로 찾아냄 |

> 💡 **비유 — 융통성 없는 사서 vs 베테랑 사서**
> - 키워드 검색 = **융통성 없는 사서.** "출산휴가"라고 정확히 말해야만 책을 줍니다. "출산 후 쉬는 거요"라고 하면 "그런 책 없습니다"라고 답합니다.
> - 의미 검색 = **베테랑 사서.** "출산 후 쉬는 거요"라고 말해도 "아, 출산전후휴가 규정 말씀이군요!" 하고 찰떡같이 알아듣고 찾아 줍니다.

### 6-5. 임베딩은 누가 만드나?

임베딩은 사람이 손으로 좌표를 정하는 게 아니라, **임베딩 모델**이라는 전용 AI가 자동으로 만들어 줍니다. 대표적인 모델들(2025~2026 기준):

- **OpenAI** `text-embedding-3-small` / `text-embedding-3-large` — 간편하고 성능 좋아 가장 널리 쓰임 (각각 1,536 / 3,072차원)
- **Cohere** `embed-v4` — 다국어(한국어 포함) 강점, 기업용
- **BGE-M3** (오픈소스, BAAI) — 무료로 직접 운영 가능, 100개 이상 언어 지원

> 🔧 실무 팁: 한국어 문서가 많다면 **한국어를 잘 처리하는 다국어 임베딩 모델**(예: Cohere embed-v4, BGE-M3, 또는 OpenAI 최신 모델)을 고르는 것이 정답률에 직접 영향을 줍니다.

> ⚠️ 중요한 규칙: **준비 단계에서 문서를 임베딩한 모델과, 응답 단계에서 질문을 임베딩하는 모델은 반드시 같아야** 합니다. 서로 다른 모델은 서로 다른 '의미 지도'를 쓰기 때문에, 같은 지도 위에서 비교해야 거리가 맞습니다.

### 🧭 Section 6 한 줄 요약

> Embedding은 단어·문장의 **'의미'를 숫자 좌표(벡터)로** 바꾸는 기술이며, **의미가 비슷하면 좌표도 가깝다.** 덕분에 글자가 달라도 '뜻이 통하면' 찾아내는 **의미 기반 검색**이 가능해진다.

## Section 7. Vector Database — 의미로 검색하는 특별한 창고

### 7-1. Vector DB의 역할

Section 6에서 모든 문서 조각을 '의미 좌표(벡터)'로 바꿨습니다. 이제 이 벡터들이 수백만 개라면, 어딘가에 **저장**하고 질문이 올 때 **가장 가까운 것을 순식간에 찾아내야** 합니다. 이 일을 전담하는 것이 **벡터 데이터베이스(Vector Database)** 입니다.

벡터 DB의 두 가지 임무:
1. **저장(Store):** 수많은 벡터 + 원문 + 메타데이터를 보관.
2. **유사도 검색(Search):** 질문 벡터가 들어오면, **의미적으로 가장 가까운 벡터 K개를 매우 빠르게** 찾아 반환.

> 💡 **비유 — 의미순으로 정리된 마법의 서가**
>
> 일반 창고는 "이름순", "날짜순"으로 정리합니다. 벡터 DB는 **"의미가 비슷한 것끼리 가까이"** 정리된 마법의 서가입니다. "출산휴가"와 의미가 가까운 카드들이 같은 구역에 모여 있어, 사서가 한달음에 달려가 꺼내 옵니다.

### 7-2. 왜 일반 DB로는 안 되는가?

우리가 아는 일반 데이터베이스(엑셀, 회원 명단 같은 것)는 **'정확히 일치(exact match)'** 검색에 최적화돼 있습니다. "이름이 '홍길동'인 행"은 순식간에 찾지만, **"이 질문과 '의미가 비슷한' 행"** 은 잘 못 찾습니다.

| 구분 | 일반 데이터베이스 | 벡터 데이터베이스 |
|------|------------------|------------------|
| 저장하는 것 | 숫자·글자·날짜 (정형 데이터) | **고차원 벡터**(의미 좌표) + 메타데이터 |
| 검색 방식 | **정확히 일치**하는가? (=, LIKE) | **얼마나 가까운가?**(유사도) |
| 잘하는 질문 | "급여 500만 원 이상 직원" | "이 질문과 의미가 통하는 문단" |
| 핵심 기술 | 인덱스(B-tree 등) | **ANN**(근사 최근접 이웃 탐색) |

> 📌 핵심 기술 한 가지만: 벡터가 수백만~수십억 개면 **전부와 일일이 거리를 비교**하면 너무 느립니다. 그래서 벡터 DB는 **ANN(Approximate Nearest Neighbor, 근사 최근접 이웃)** 이라는 기법으로, "완벽하진 않아도 충분히 가까운 것들을 매우 빠르게" 찾습니다. **정확도를 약간 양보하고 속도를 크게 얻는** 영리한 타협입니다.

> 💡 비유: 도서관에서 '딱 한 권의 정답'을 위해 100만 권을 다 뒤지는 대신, **이미 주제별로 잘 정리된 구역으로 바로 가서** 그 근처만 빠르게 훑는 것과 같습니다.

### 7-3. 대표 벡터 DB 비교 (2025~2026)

> ⚠️ 먼저 짚을 점: **FAISS는 엄밀히 'DB'라기보다 '라이브러리'** 입니다. 검색 기능은 매우 빠르지만, 데이터 관리·운영 기능은 직접 챙겨야 합니다. 나머지는 운영 기능을 갖춘 '제품/서비스'에 가깝습니다.

| 이름 | 유형 | 한 줄 특징 | 강점 | 주의점 | 적합한 경우 |
|------|------|-----------|------|--------|------------|
| **FAISS** | 오픈소스 라이브러리 (Meta) | 순수 검색 속도 최강 | 매우 빠름, GPU 가속, 무료 | DB 운영기능 없음, 직접 튜닝 필요 | 연구·프로토타입, 소규모 내장형 |
| **Chroma** | 오픈소스 | 가장 쉬운 입문용 | 설치·사용 간단, 개발자 친화 | 초대규모엔 부적합 | 학습·프로토타입·소규모 서비스 |
| **Milvus** | 오픈소스 (분산) | 대규모 처리의 강자 | 수십억 벡터, 높은 처리량, 분산 | 운영 복잡도 높음 | 대용량·대규모 트래픽 |
| **Weaviate** | 오픈소스 | 하이브리드 검색 강자 | 의미+키워드 결합, 필터링 우수 | 학습 곡선 존재 | 의미+조건검색 함께 필요할 때 |
| **Pinecone** | 관리형(클라우드) | 가장 손 안 가는 완전관리형 | 자동 확장, 운영 부담 0, 안정적 | 유료·외부 클라우드 의존 | 빠른 출시, 운영 인력 최소화 |

> 🔧 그 외 자주 보이는 선택지
> - **Qdrant** — Rust 기반 고성능, 조건 필터링 검색에 강함.
> - **pgvector** — 기존 PostgreSQL DB에 벡터 검색을 더하는 확장. **이미 PostgreSQL을 쓰는 회사**가 별도 DB 없이 시작하기에 좋음.

> 🔧 **선택 가이드 (비개발자 의사결정용)**
> - **"일단 배우고 테스트"** → Chroma 또는 FAISS
> - **"운영은 맡기고 빨리 출시"** → Pinecone
> - **"대규모·고성능, 직접 운영 역량 있음"** → Milvus / Qdrant
> - **"의미검색 + 조건(부서·날짜)필터 동시에"** → Weaviate
> - **"이미 PostgreSQL 사용 중"** → pgvector
>
> 정답은 없습니다. **데이터 규모, 운영 인력, 예산, 기존 시스템**에 맞춰 고르면 됩니다.

### 🧭 Section 7 한 줄 요약

> 벡터 DB는 **'의미 좌표를 저장하고, 가장 가까운 것을 초고속으로 찾아주는' 특수 창고**다. 일반 DB가 '정확히 일치'를 찾는다면, 벡터 DB는 **'얼마나 비슷한가'** 를 찾는다.

<a id="section-8"></a>
## Section 8. Retrieval — 질문에 가장 가까운 조각 찾기

여기서부터는 **응답 단계(실시간)** 입니다. 사용자가 드디어 질문을 던졌습니다.

### 8-1. Retrieval의 전체 과정

```
사용자 질문: "출산휴가는 며칠인가요?"
        │
        ▼  ① 질문을 '같은 임베딩 모델'로 벡터화
   질문 벡터: [0.12, -0.55, 0.31, ...]
        │
        ▼  ② 벡터 DB에서 '가장 가까운' 청크들을 검색 (유사도 비교)
   후보들:  청크A(유사도 0.92) · 청크B(0.61) · 청크C(0.40) ...
        │
        ▼  ③ 가장 가까운 상위 K개만 선택 (Top-K)
   선택:   청크A, 청크B   (K=2)
```

> 📌 핵심: 질문도 **문서와 똑같은 임베딩 모델**로 벡터로 바꾼 뒤, **같은 의미 지도** 위에서 가까운 이웃을 찾습니다. (Section 6의 마지막 ⚠️와 연결)

### 8-2. Similarity Search (유사도 검색)

**유사도 검색**은 "질문 벡터와 의미적으로 가장 가까운 문서 벡터를 찾는 것"입니다. Section 6의 '의미 지도'에서, 질문이라는 점을 찍고 **그 주변에 모여 있는 문서 점들**을 집어 오는 것이죠.

> 💡 비유: 지도에 내 위치(질문)를 찍고 **"가장 가까운 카페(관련 문서) 3곳 찾기"** 버튼을 누르는 것과 같습니다.

### 8-3. Cosine Similarity (코사인 유사도) — 직관적으로

두 벡터가 얼마나 비슷한지 재는 가장 흔한 잣대가 **코사인 유사도**입니다. 수학을 빼고 직관만 봅니다.

> 💡 **비유 — 두 화살표가 같은 방향을 가리키는가?**
>
> 각 벡터를 **원점에서 뻗어 나가는 화살표**라고 생각하세요. 코사인 유사도는 **두 화살표 사이의 '방향(각도)'이 얼마나 비슷한가**를 봅니다. 길이(크기)가 아니라 **방향**이 핵심입니다.

```
유사도 = 1.0   →→  →→      두 화살표가 같은 방향   = 의미가 매우 비슷 ✅
유사도 ≈ 0.0   →   ↑        두 화살표가 직각        = 거의 무관 😐
유사도 = -1.0  →→  ←←      두 화살표가 반대 방향   = 의미가 정반대 ❌
```

- **1에 가까울수록** 두 의미가 **비슷**합니다.
- 0에 가까우면 **관련 없음**, -1에 가까우면 **정반대**입니다.
- 텍스트 임베딩에서는 보통 값이 **0과 1 사이**에 분포하며, **1에 가까운 것을 고르면** 됩니다.

> ⚠️ 수학적으로 코사인 유사도의 이론적 범위는 -1~1이지만, 실무에서 텍스트 임베딩끼리 비교하면 대개 양수 영역(0~1)에 모입니다. 비개발자는 **"1에 가까울수록 비슷하다"** 만 기억하면 충분합니다.

### 8-4. Top-K Retrieval (상위 K개 검색)

검색하면 유사도 순으로 줄을 세울 수 있습니다. 이 중 **가장 비슷한 상위 K개만** 골라 쓰는 것을 **Top-K 검색**이라고 합니다. (K는 우리가 정하는 숫자: 보통 3~10개)

왜 전부 안 쓰고 K개만? → LLM 책상(컨텍스트)은 유한하고, **너무 많이 넣으면 잡음이 섞여 오히려 답이 흐려지기** 때문입니다.

> 💡 비유: 사서가 관련 책 1만 권을 다 가져오면 압도당합니다. **"가장 관련 깊은 3권만"** 뽑아 줘야 일하기 좋습니다.

> 🔧 실무 팁: K가 **너무 작으면** 정작 필요한 근거를 놓치고(누락), **너무 크면** 잡음·비용이 늘고 'Lost in the middle'이 생깁니다. **보통 3~5에서 시작**해 조정합니다.

### 🧭 Section 8 한 줄 요약

> Retrieval은 **질문을 벡터로 바꿔, 의미 지도에서 가장 가까운 문서 조각 Top-K개를 찾아오는** 단계다. '가까움'은 주로 **코사인 유사도(방향이 얼마나 같은가)** 로 잰다.

<a id="section-9"></a>
## Section 9. Prompt Augmentation — AI에게 '컨닝페이퍼'를 쥐여주기

검색으로 관련 조각을 찾았습니다. 이제 이 조각들을 **질문과 함께 잘 묶어서 LLM에게 전달**해야 합니다. 이 '묶는' 과정이 **Prompt Augmentation(프롬프트 보강)** 이며, RAG의 'A(Augmented)'에 해당합니다.

> 💡 **비유 — 오픈북 시험의 마지막 손질**
>
> 똑똑한 학생(LLM)에게 그냥 "출산휴가 며칠?"이라고 물으면 기억에 의존해 틀릴 수 있습니다.
> 대신 **"여기 회사 규정 발췌본이야. 이것만 근거로, 출산휴가가 며칠인지 답해 줘"** 라며 자료를 함께 건네면, 학생은 그 자료를 보고 정확히 답합니다. 이 '자료 + 지시문 + 질문'을 한 장으로 합치는 것이 프롬프트 보강입니다.

### 9-1. 프롬프트 보강의 구조

보강된 프롬프트는 보통 세 부분으로 구성됩니다.

1. **지시(Instruction):** AI가 지켜야 할 규칙. (예: "아래 자료만 근거로 답하라. 없으면 모른다고 하라.")
2. **컨텍스트(Context):** 검색해 온 문서 조각들. (= 컨닝페이퍼)
3. **질문(Question):** 사용자의 원래 질문.

### 9-2. 단계별 예제

#### ① 사용자 질문
```
출산휴가는 며칠인가요?
```

#### ② 검색 결과 (Retrieval로 찾아온 조각들)
```
[청크 A] (출처: 사내복지규정.pdf, 7쪽)
"제12조(출산전후휴가) 임신 중의 여성에게는 출산 전후를 통하여 90일의
 출산전후휴가를 부여한다. 다태아를 임신한 경우에는 120일로 한다."

[청크 B] (출처: 사내복지규정.pdf, 7쪽)
"출산전후휴가 중 최초 60일은 유급으로 한다."
```

#### ③ 최종 프롬프트 (보강 완료 — LLM에게 실제로 들어가는 글)
```
당신은 사내 규정 안내 도우미입니다.
아래 [참고 자료]에 있는 내용만 근거로 답하세요.
자료에 없는 내용은 추측하지 말고 "규정에서 찾을 수 없습니다"라고 답하세요.
답변 끝에는 근거가 된 출처를 표기하세요.

[참고 자료]
- (사내복지규정.pdf 7쪽) 제12조(출산전후휴가) 임신 중의 여성에게는 출산
  전후를 통하여 90일의 출산전후휴가를 부여한다. 다태아 임신 시 120일.
- (사내복지규정.pdf 7쪽) 출산전후휴가 중 최초 60일은 유급으로 한다.

[질문]
출산휴가는 며칠인가요?

[답변]
```

#### ④ LLM 응답
```
출산전후휴가는 출산 전후를 통틀어 총 90일이 부여됩니다.
다태아를 임신한 경우에는 120일입니다. 이 중 최초 60일은 유급입니다.

(출처: 사내복지규정.pdf 7쪽, 제12조)
```

### 9-3. 왜 이게 강력한가?

- **환각 방지:** "자료에 없으면 모른다고 하라"는 지시 + 실제 근거 자료 → 지어내기 차단.
- **출처 제공:** 자료에 출처를 함께 넣어 줬으니 답변도 출처를 댑니다.
- **최신성:** 자료(청크)만 바꾸면 답도 바뀝니다. 모델은 그대로 둬도 됩니다.

> ⚠️ 주의: 보강 프롬프트의 **지시문이 부실하면** 모델이 자료를 무시하고 멋대로 답할 수 있습니다. "**제공된 자료만** 사용하라", "**없으면 모른다고** 하라" 같은 문구가 RAG 품질에 생각보다 큰 영향을 줍니다.

### 🧭 Section 9 한 줄 요약

> Prompt Augmentation은 **검색해 온 근거 자료 + 행동 지시 + 사용자 질문**을 한 장의 프롬프트로 합쳐 LLM에게 건네는 단계다. 이 '컨닝페이퍼' 덕분에 LLM이 **근거 있고 출처 있는** 답을 한다.

<a id="section-10"></a>
## Section 10. End-to-End 예제 — "출산휴가는 며칠인가요?"

이제까지 배운 모든 조각을 하나로 꿰어, **처음부터 끝까지 한 번에** 따라가 봅니다.

### 무대 설정

- **문서:** `사내복지규정.pdf` (20쪽)
- **사용자:** 인사팀 김대리
- **질문:** "출산휴가는 며칠인가요?"

---

### 【준비 단계 — 미리 해 둔 일】

#### STEP 1. 원본 문서 (일부 발췌)
```
... (5쪽) 제10조(연차휴가) 1년간 80% 이상 출근한 근로자에게 15일의 유급휴가를 준다. ...
... (7쪽) 제12조(출산전후휴가) 임신 중의 여성에게는 출산 전후를 통하여 90일의
         출산전후휴가를 부여한다. 다태아를 임신한 경우에는 120일로 한다.
         출산전후휴가 중 최초 60일은 유급으로 한다. ...
... (9쪽) 제15조(경조사 휴가) 본인 결혼 5일, 배우자 출산 10일을 부여한다. ...
```

#### STEP 2. Chunking (조각내기)
규정을 조항 단위로 쪼갰습니다. (Recursive 방식 가정)
```
청크 #1 = "제10조(연차휴가) 1년간 80% 이상 출근한 근로자에게 15일의 유급휴가를 준다."
청크 #2 = "제12조(출산전후휴가) ... 90일 ... 다태아 120일 ... 최초 60일 유급."
청크 #3 = "제15조(경조사 휴가) 본인 결혼 5일, 배우자 출산 10일."
```

#### STEP 3. Embedding (벡터화)
각 청크를 임베딩 모델에 넣어 의미 좌표로 바꿉니다. (실제론 숫자 수천 개; 여기선 상징적으로 표기)
```
청크 #1 → [0.10, 0.83, ...]   (주제: 연차)
청크 #2 → [0.77, 0.12, ...]   (주제: 출산휴가)
청크 #3 → [0.31, 0.55, ...]   (주제: 경조사)
```

#### STEP 4. Indexing (벡터 DB에 저장)
```
┌─────────── Vector DB ───────────┐
│ #1 [0.10,0.83,...]  연차  (5쪽)  │
│ #2 [0.77,0.12,...]  출산  (7쪽)  │
│ #3 [0.31,0.55,...]  경조사(9쪽)  │
└─────────────────────────────────┘
   ※ 여기까지가 '미리 준비'. 질문 전에 완료됨.
```

---

### 【응답 단계 — 질문이 들어온 순간】

#### STEP 5. 질문 임베딩
김대리의 질문을 **같은 임베딩 모델**로 벡터화합니다.
```
"출산휴가는 며칠인가요?" → [0.74, 0.15, ...]  (질문 벡터)
```

#### STEP 6. Retrieval (유사도 검색 + Top-K)
질문 벡터와 각 청크의 코사인 유사도를 계산합니다.
```
질문 [0.74, 0.15, ...] 와의 유사도:
   청크 #2 (출산)   → 0.95  ✅ 가장 가까움!
   청크 #3 (경조사) → 0.42
   청크 #1 (연차)   → 0.18

→ Top-K(K=1~2) 선택: 청크 #2  (필요시 #3까지)
```
> 📌 주목: 질문에는 "출산휴가", 문서에는 "출산전후휴가". **글자는 다르지만 의미가 가까워** 정확히 찾아냈습니다. (의미 기반 검색의 힘)

#### STEP 7. Prompt Augmentation (프롬프트 생성)
```
아래 [참고 자료]만 근거로 답하세요. 없으면 "규정에서 찾을 수 없습니다"라고 하세요.
출처를 함께 표기하세요.

[참고 자료]
- (사내복지규정.pdf 7쪽, 제12조) 임신 중의 여성에게 출산 전후 통산 90일의
  출산전후휴가를 부여한다. 다태아는 120일. 최초 60일은 유급.

[질문] 출산휴가는 며칠인가요?
```

#### STEP 8. Answer (LLM이 근거를 보고 생성)
```
출산전후휴가는 출산 전후를 합쳐 총 90일이 부여됩니다.
다태아를 임신한 경우에는 120일이며, 이 중 최초 60일은 유급입니다.
(출처: 사내복지규정.pdf 7쪽, 제12조)
```

### 전체 추적 요약

```
질문 "출산휴가 며칠?"
   └─[임베딩]→ 질문 벡터
        └─[검색]→ 청크#2(출산, 유사도0.95) 선택
             └─[보강]→ "이 자료만 보고 답해" + 청크#2 + 질문
                  └─[LLM]→ "총 90일입니다 (출처: 7쪽)"  ✅
```

> 📌 만약 RAG가 없었다면? LLM은 우리 회사 규정을 모르므로 "보통 90일 정도입니다(일반론)" 같은 **출처 없는, 회사 규정과 다를 수 있는** 답을 했을 것입니다. RAG는 이를 **우리 회사 7쪽에 근거한 답**으로 바꿔 줍니다.

### 🧭 Section 10 한 줄 요약

> 같은 임베딩 모델로 **문서도 질문도 좌표로 바꾸고**, **가장 가까운 조각을 찾아**, **근거로 함께 넣어** 답하게 하면 — 글자가 달라도 정확히, 출처와 함께 답한다. 이것이 RAG의 전부다.

<a id="section-11"></a>
## Section 11. Advanced RAG — 한 단계 더 똑똑하게

지금까지 배운 것은 **기본형 RAG(Naive RAG)** 입니다. 현업에서는 정확도를 더 끌어올리려고 여러 기법을 얹습니다. 여기서는 **'무엇인지'만 가볍게** 소개합니다. (세부 구현은 몰라도 됩니다.)

| 기법 | 한 줄 설명 | 비유 |
|------|-----------|------|
| **Hybrid Search** (하이브리드 검색) | **의미 검색(벡터) + 키워드 검색(정확 일치)** 을 함께 사용. 제품번호·법조항처럼 정확한 단어가 중요한 질문에 강함. | 베테랑 사서 + 융통성 없는 사서를 **둘 다** 부르기 |
| **Re-ranking** (재정렬) | 1차로 넉넉히 가져온 뒤, 더 정밀한 모델로 **다시 점수 매겨 최종 순위**를 정함. 정확도 크게 향상. | 서류 통과자 중 **면접으로 최종 합격자**를 가림 |
| **Metadata Filtering** (메타데이터 필터링) | "2024년 이후", "인사팀 문서"처럼 **조건으로 먼저 거른 뒤** 의미 검색. 권한·기간 제어에 유용. | 도서관에서 **'특정 서가'로 범위를 좁힌 뒤** 찾기 |
| **Parent Document Retrieval** (부모 문서 검색) | 검색은 **작은 조각**으로 정확히 하되, LLM에는 그 조각이 속한 **더 큰 원문(부모)** 을 넘겨 문맥 보강. | 색인 카드로 찾고, **그 카드가 가리키는 책의 해당 장 전체**를 펼쳐 줌 |
| **Multi-query Retrieval** (다중 질의 검색) | 질문 하나를 **여러 표현으로 바꿔** 각각 검색한 뒤 합침. 누락 줄이고 재현율↑. | 같은 질문을 **여러 사서에게 다르게 물어** 결과 합치기 |
| **Graph RAG** (그래프 RAG) | 문서에서 **개체·관계를 뽑아 지식 그래프**로 만든 뒤 검색. "여러 문서에 흩어진 정보를 연결"하는 복합 질문에 강함. | 인물·사건이 **선으로 연결된 관계도**를 보고 추론 |
| **Agentic RAG** (에이전트형 RAG) | LLM이 **스스로 계획**을 세워 검색을 여러 번 반복하고, 필요하면 계산기·DB·API 같은 **도구도 호출**하며 답을 다듬음. | 수동적 학생이 아니라, **스스로 조사하는 연구원** |

> 🔧 **2025~2026 동향 메모**
> - 현장은 기본형 → **Advanced RAG(하이브리드·재정렬)** → **Agentic RAG**로 진화 중이며, 최근엔 질문 난이도에 따라 검색 전략을 **자동으로 바꾸는 Adaptive RAG**도 주목받습니다.
> - Graph RAG 계열은 비용이 관건이었는데, Microsoft의 **LazyGraphRAG**, 오픈소스 **LightRAG·HippoRAG 2** 등으로 **비용·지연을 크게 낮춘** 변형이 등장했습니다.
> - 한 사례에서는 GraphRAG 도입으로 검색 정확도 지표(MRR)가 약 77% 향상되고, 상담사의 평균 처리 시간이 약 28% 단축됐다고 보고되기도 했습니다.

> ⚠️ 오해 방지: **처음부터 이 기법들을 다 넣을 필요는 없습니다.** 기본형 RAG로 시작해 측정해 보고, **부족한 부분만** 골라 얹는 것이 정석입니다. (과한 설계는 비용·복잡도만 키웁니다.)

### 🧭 Section 11 한 줄 요약

> Advanced RAG는 기본형에 **하이브리드 검색·재정렬·필터링·그래프·에이전트** 등을 더해 정확도를 끌어올리는 기법들이다. **필요할 때 하나씩** 도입하는 것이 핵심.

<a id="section-12"></a>
## Section 12. 실제 기업 활용 사례 (2025–2026)

RAG는 이론이 아니라 **이미 현장에서 돈을 벌고 비용을 줄이는** 기술입니다. 시장 규모도 빠르게 큽니다. (전 세계 RAG 시장은 2024년 약 12억 달러에서 2030년 약 110억 달러로, 연 49% 안팎의 고성장이 전망됩니다.) 분야별로 봅니다.

### 12-1. 사내 지식 검색 (Enterprise Search) — 가장 큰 시장

- **문제:** 직원이 규정·매뉴얼·위키를 찾느라 하루에도 수십 분씩 허비. "그 문서 어디 있더라?"
- **RAG 해법:** 흩어진 사내 문서를 RAG로 묶어, **자연어로 물으면 출처와 함께 요약 답변**.
- **효과:** 도입 기업들에서 **정보 검색 시간 평균 70% 이상 단축** 사례 보고. 직원은 단순 검색에서 해방돼 고부가가치 업무에 집중.
- **사례:** 삼성SDS는 기업 맞춤형 RAG 구축 사례와 방법론을 공유하고 있으며, 한화그룹의 사내 챗봇 'AIDA'는 건설 분야 법규(하도급법·중대재해처벌법 등)와 업무 노하우를 즉시 검색·제공하는 것으로 알려져 있습니다.

### 12-2. 고객센터 챗봇 (Customer Support)

- **문제:** 반복 문의가 상담사를 소진시키고, 야간·주말 응대가 어려움.
- **RAG 해법:** 제품 매뉴얼·FAQ·정책 문서를 근거로 **정확하고 출처 있는** 답변을 24시간 자동 제공. 모르면 "상담사 연결".
- **효과:** 고객이 **스스로 문제를 해결**하는 비율 상승, 상담사는 복잡한 문의에 집중. (한 클라우드 기술지원 사례에서 셀프 해결 가능 케이스가 약 68%로 분석되어 자동화 효과가 큼.)

### 12-3. 법률 (Legal)

- **문제:** 방대한 판례·계약서·규정에서 **정확한 조항과 근거**를 찾아야 하고, 틀리면 치명적.
- **RAG 해법:** 판례·계약 저장소를 의미 검색하고, **답변에 정확한 조항·판례를 인용**. (출처 제시가 필수인 분야와 궁합이 좋음.)
- **사례:** 해외 스타트업이 지식그래프 RAG로 **법률 문서 분석 정확도 92%** 달성 보고; 국내 리걸테크는 결정문·심결례·판례집 등 **4,360쪽 분량**을 학습한 변호사용 AI 챗봇 공개.

### 12-4. 금융 (Finance)

- **문제:** 분석가가 수십 개 자료(시장 데이터, 재무보고서, 내부 지표)를 모아 보고서를 손으로 작성.
- **RAG 해법:** 관련 자료를 자동으로 모아 **맞춤형 요약·리스크 분석**을 생성. 규정 준수를 위해 **출처 추적**이 가능.
- **사례:** KB국민카드가 LLM 기반 질의응답 챗봇에 기업 이벤트 정보를 API로 연동하는 방식으로 RAG를 구축.

### 12-5. 제조 (Manufacturing)

- **문제:** 장비 매뉴얼·작업표준(SOP)·고장 이력이 방대하고, 현장에서 **즉시** 찾아야 함.
- **RAG 해법:** 매뉴얼·정비 기록을 RAG로 묶어, 현장 작업자가 **"○○ 에러코드 조치법"** 을 자연어로 즉시 조회. 안전·품질 규정 안내에도 활용.
- **효과:** 신입·교대 인력의 숙련 속도 향상, 다운타임 단축, 베테랑의 암묵지 보존.

### 12-6. 의료 (Healthcare)

- **문제:** 진료 지침·논문·약물 정보가 끊임없이 갱신되고 방대함. 정확성과 근거가 생명.
- **RAG 해법:** 최신 가이드라인·문헌을 근거로 **임상 의사결정 지원**, 의료진 행정·문서 업무 보조.
- **주의:** 의료는 **반드시 출처 제시와 전문가 검수**가 전제. RAG는 '대체'가 아니라 '보조' 도구로 설계해야 함. (한 임상 의사결정 연구에서, 주제 경계에 맞춘 적응형 청킹이 고정 분할 대비 정확도를 크게 끌어올린 결과도 보고됨.)

### 분야별 요약표

| 분야 | 대표 활용 | 핵심 가치 |
|------|----------|-----------|
| 사내 지식 검색 | 규정·매뉴얼·위키 통합 검색 | 검색시간 단축, 생산성 |
| 고객센터 | FAQ·매뉴얼 기반 자동 응대 | 24시간 응대, 셀프 해결률↑ |
| 법률 | 판례·계약 검색 + 조항 인용 | 정확성, 출처 제시 |
| 금융 | 자료 통합·요약·리스크 분석 | 시간 절약, 규정 준수 |
| 제조 | 매뉴얼·정비이력 즉시 조회 | 다운타임↓, 암묵지 보존 |
| 의료 | 가이드라인 기반 의사결정 지원 | 최신성, 근거 기반(전문가 검수 전제) |

> 📌 공통 패턴: **(1) 문서가 많고 (2) 정답이 문서 안에 있으며 (3) 틀리면 곤란하고 (4) 출처가 중요한** 업무일수록 RAG의 효과가 큽니다.

### 🧭 Section 12 한 줄 요약

> RAG는 사내 검색·고객센터·법률·금융·제조·의료 등 **'문서 많고, 정확성·출처가 중요한'** 모든 분야에서 이미 실전 도입되어 검색시간 단축·정확도 향상 등 측정 가능한 성과를 내고 있다.

<a id="section-13"></a>
## Section 13. RAG vs Fine-tuning — 언제 무엇을 쓸까

가장 많이 받는 질문입니다. **"둘 중 뭘 써야 하나요?"** 결론부터: **대체로 RAG가 1순위**이고, 둘은 **경쟁이 아니라 보완** 관계입니다.

### 13-1. 핵심 비교표

| 항목 | RAG (검색 증강) | Fine-tuning (미세조정) |
|------|----------------|------------------------|
| **목적** | 모델에 **'지식'(무엇을 아는가)** 을 외부에서 공급 | 모델의 **'행동·말투·형식'(어떻게 하는가)** 을 조정 |
| **비용** | 상대적으로 낮음 (문서만 준비) | 높음 (데이터 구축 + 학습 자원) |
| **최신성** | **매우 우수** — 문서만 바꾸면 즉시 반영 | **약함** — 새 지식 반영하려면 재학습 필요 |
| **유지보수** | 쉬움 — 문서 추가/수정/삭제로 관리 | 어려움 — 변경 때마다 재학습·재배포 |
| **정확성/근거** | **출처 인용 가능**, 환각 적음 | 출처 제시 불가, 지식은 여전히 '기억'에 의존 |
| **구현 난이도** | 중간 (파이프라인 구성 필요) | 높음 (ML 전문성·데이터 품질 요구) |
| **데이터 요구** | 문서(비정형 텍스트)면 충분 | 양질의 학습용 예시 데이터셋 필요 |
| **적합한 경우** | 자주 바뀌는 사실, 사내 문서, 출처 필요 | 일관된 말투/형식, 특수 도메인 어투, 특정 출력 양식 |

### 13-2. 비유로 한 번에

> 💡 **신입사원에게 일을 시키는 두 가지 방법**
> - **RAG = 업무 매뉴얼을 책상에 비치하고 "필요하면 찾아보고 답해"** → 매뉴얼만 갱신하면 항상 최신. 어디서 봤는지도 말해 줌.
> - **Fine-tuning = 신입을 몇 달간 연수원에 보내 '우리 회사 사람'으로 체화시키기** → 말투·태도·업무 스타일이 몸에 뱀. 하지만 어제 바뀐 규정은 모름(또 연수 필요).

### 13-3. 의사결정 가이드

```
질문: 내 문제는 '지식'인가 '행동(스타일)'인가?

 ├─ "최신·정확한 사실/문서 기반 답변이 필요" ─────────────►  RAG ✅
 │     (예: 사내규정 Q&A, 고객지원, 법률·금융 검색)
 │
 ├─ "특정 말투/출력형식/도메인 어투를 일관되게" ──────────►  Fine-tuning
 │     (예: 항상 우리 브랜드 톤, 정해진 JSON 양식으로 출력)
 │
 └─ "둘 다 필요" ───────────────────────────────────────►  RAG + Fine-tuning 병행
       (지식은 RAG로, 말투·형식은 Fine-tuning으로)
```

> 📌 실무 정석: **"먼저 RAG로 시작하라."** 대부분의 기업 요구(사내 지식 활용, 최신성, 출처)는 RAG로 해결됩니다. 그래도 말투·형식이 부족하면 그때 Fine-tuning을 **추가**하세요. 처음부터 Fine-tuning으로 가는 것은 대개 과합니다.

> ⚠️ 다시 강조: Fine-tuning은 **'지식 주입기'가 아닙니다.** 자주 바뀌는 사실을 Fine-tuning으로 넣으려는 시도는 비싸고, 환각도 못 막고, 출처도 못 댑니다. 그 일은 RAG의 몫입니다.

### 🧭 Section 13 한 줄 요약

> **지식·최신성·출처가 중요하면 RAG, 말투·형식·행동을 바꾸려면 Fine-tuning.** 둘은 경쟁이 아니라 보완이며, **대부분 RAG부터 시작**하는 것이 정답이다.

## Section 14. Python 구현 예제 — 코드로 보는 RAG

> 비개발자 여러분, **코드를 외우거나 직접 짤 필요는 전혀 없습니다.** 목표는 **"RAG가 코드로는 이런 모양이구나"** 하는 감(感)을 잡는 것입니다. 각 코드 묶음 아래에 **"이 코드가 하는 일"** 을 일상어로 풀어 두었으니 그것만 따라 읽으세요.
>
> 🔧 두 예제 모두 OpenAI API 키가 필요합니다. (`OPENAI_API_KEY` 환경변수로 설정) 코드는 2025~2026 기준 최신 LangChain 스타일로 작성했습니다.

### 예제 1. 최소 구현 — OpenAI + LangChain + FAISS

> "RAG가 정말 이렇게 짧아?" 싶을 만큼 핵심만 담은 버전입니다.

```python
# pip install langchain langchain-openai langchain-community faiss-cpu

from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# 0) 우리 회사 문서 (지금은 간단히 문자열 몇 개로 대신)
texts = [
    "출산전후휴가는 출산 전후를 통하여 총 90일이 부여된다. 다태아는 120일이다.",
    "연차휴가는 1년간 80% 이상 출근 시 15일이 부여된다.",
    "경조사 휴가 중 본인 결혼은 5일, 배우자 출산은 10일이다.",
]

# 1) 임베딩 모델 준비 + 2) 벡터DB(FAISS)에 저장
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_texts(texts, embeddings)

# 3) 검색기: 질문과 가장 비슷한 조각 2개를 찾도록 설정
retriever = vectorstore.as_retriever(search_kwargs={"k": 2})

# 4) 프롬프트: '자료를 보고 답하라'는 지시 + 자료 자리 + 질문 자리
prompt = ChatPromptTemplate.from_template(
    "아래 자료만 근거로 질문에 답하세요. 없으면 모른다고 하세요.\n\n"
    "자료:\n{context}\n\n질문: {question}"
)

# 5) 답변 생성 LLM
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 검색된 조각들을 하나의 글로 합치는 도우미
def format_docs(docs):
    return "\n".join(d.page_content for d in docs)

# 6) 전체 RAG 흐름을 하나로 연결: 검색 → 프롬프트 → LLM → 텍스트
chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 7) 질문!
print(chain.invoke("출산휴가는 며칠인가요?"))
# 예상 출력: 출산전후휴가는 총 90일이며, 다태아의 경우 120일입니다.
```

> **이 코드가 하는 일 (한 문장씩)**
> - `texts` : 우리 회사 규정을 (지금은 간단히) 문자열로 준비합니다. = **Document/Chunk**
> - `OpenAIEmbeddings` : 글을 **의미 좌표(벡터)로 바꾸는 변환기**를 준비합니다. = **Embedding**
> - `FAISS.from_texts(...)` : 그 좌표들을 **검색 가능한 창고에 저장**합니다. = **Vector DB / Indexing**
> - `as_retriever(k=2)` : "질문과 **가장 비슷한 2조각**을 찾아 줘"라는 **검색기**를 만듭니다. = **Retrieval**
> - `prompt` : "**이 자료만 보고** 답해"라는 **지시문 틀**입니다. = **Prompt Augmentation**
> - `llm` : 실제로 답 문장을 **써 주는 AI**입니다. = **LLM** (`temperature=0`은 "창의적 변형을 최소화하고 **매번 일관되고 안정적인 답**을 내라"는 설정. 사실 위주 답변에 적합)
> - `chain` : 위 단계들을 **검색→자료합치기→프롬프트→LLM 순서로 연결한 컨베이어 벨트**입니다.
> - `chain.invoke(...)` : 질문을 벨트에 올리면 **출처에 근거한 답**이 나옵니다.

### 예제 2. 실무 스타일 — 진짜 파일(PDF)로 RAG 구축

실제 업무처럼 **PDF 파일**을 읽고, 정식 단계(Loader → Splitter → Embedding → Vector Store → Retriever → PromptTemplate)를 모두 갖춘 버전입니다.

```python
# pip install langchain langchain-openai langchain-community faiss-cpu pypdf

from langchain_community.document_loaders import PyPDFLoader          # Loader
from langchain_text_splitters import RecursiveCharacterTextSplitter  # Splitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI            # Embedding & LLM
from langchain_community.vectorstores import FAISS                    # Vector Store
from langchain_core.prompts import ChatPromptTemplate                # PromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# ── [준비 단계: 문서를 검색 가능하게 만들기] ────────────────────────

# 1) Loader: PDF 파일을 페이지 단위로 읽어 들인다
loader = PyPDFLoader("사내복지규정.pdf")
documents = loader.load()

# 2) Splitter: 긴 페이지를 적당한 크기(500자, 50자 겹침)로 쪼갠다 (Recursive)
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 한 조각의 최대 길이
    chunk_overlap=50,      # 조각 사이 겹치는 길이(문맥 보존)
)
chunks = splitter.split_documents(documents)

# 3) Embedding + 4) Vector Store: 조각을 벡터로 바꿔 FAISS에 저장
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(chunks, embeddings)

# (선택) 다음엔 다시 안 만들도록 디스크에 저장해 둘 수 있다
# vectorstore.save_local("faiss_index")

# ── [응답 단계: 질문에 답하기] ───────────────────────────────────

# 5) Retriever: 질문과 가장 비슷한 조각 4개를 가져오도록 설정
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# 6) PromptTemplate: 역할·규칙·자료·질문을 한 장으로 구성
prompt = ChatPromptTemplate.from_template(
    """당신은 사내 규정 안내 도우미입니다.
아래 [참고 자료]에 있는 내용만 근거로 답변하세요.
자료에 없으면 추측하지 말고 "규정에서 찾을 수 없습니다"라고 답하세요.
가능하면 근거가 된 부분을 함께 알려 주세요.

[참고 자료]
{context}

[질문]
{question}

[답변]"""
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def format_docs(docs):
    # 각 조각의 본문을 줄바꿈 두 번으로 이어 붙인다
    return "\n\n".join(d.page_content for d in docs)

# 전체 파이프라인 연결 (검색 → 프롬프트 → LLM → 문자열)
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 사용
question = "출산휴가는 며칠인가요?"
answer = rag_chain.invoke(question)
print(answer)
```

> **이 코드가 하는 일 (단계별)**
> - **1) Loader (`PyPDFLoader`)** — `사내복지규정.pdf`를 열어 **글자를 읽어 들입니다.** Word·웹페이지·노션 등은 각각 다른 로더를 끼우면 됩니다. (= Document Loading)
> - **2) Splitter (`RecursiveCharacterTextSplitter`)** — 긴 내용을 **500자짜리 조각으로, 50자씩 겹쳐서** 쪼갭니다. 겹침(overlap)은 경계에서 문맥이 끊기지 않게 하는 장치입니다. (= Chunking)
> - **3) Embedding (`OpenAIEmbeddings`)** — 각 조각을 **의미 좌표(벡터)** 로 바꿉니다. (= Embedding)
> - **4) Vector Store (`FAISS.from_documents`)** — 그 벡터들을 **검색 가능한 창고에 저장**합니다. `save_local`로 저장해 두면 다음엔 재계산 없이 불러올 수 있어 비용·시간을 아낍니다. (= Indexing)
> - **5) Retriever (`as_retriever(k=4)`)** — 질문이 오면 **가장 비슷한 4조각**을 찾아오는 검색기입니다. (= Retrieval)
> - **6) PromptTemplate** — "**자료만 근거로, 없으면 모른다고, 출처도**"라는 **규칙이 담긴 틀**입니다. 이 지시문이 답변 품질(환각 억제·출처 표기)을 크게 좌우합니다. (= Prompt Augmentation)
> - **`rag_chain`** — 위 모든 단계를 **하나의 흐름으로 연결**한 것. `invoke("질문")` 한 줄이면 **검색 → 근거 결합 → 답변 생성**이 자동으로 일어납니다.

> 🔧 **실무에서 더 신경 쓰는 것들 (참고만)**
> - 문서가 바뀌면 해당 부분만 다시 인제스천(증분 업데이트).
> - 답변에 **실제 출처(파일명·페이지)** 를 함께 노출해 신뢰도↑ (메타데이터 활용).
> - 정답률이 낮으면 → 청크 크기 조정, K 조정, **Re-ranking·Hybrid Search**(Section 11) 추가.
> - **평가(Evaluation):** 예상 질문-정답 세트를 만들어 정답률을 정기적으로 측정.
> - **보안:** 사용자 권한에 따라 검색 대상 문서를 제한(메타데이터 필터링).

### 🧭 Section 14 한 줄 요약

> 코드로 본 RAG도 결국 **Loader → Splitter → Embedding → Vector Store → Retriever → Prompt → LLM** 의 한 줄 흐름이다. LangChain은 이 조각들을 **레고처럼 끼워 맞추는** 도구일 뿐이다.

## 부록

### 부록 A. RAG가 만능은 아니다 — 한계와 주의점

> 균형 잡힌 이해를 위해, RAG의 약점도 솔직히 짚습니다.

| 한계 | 설명 | 대응 |
|------|------|------|
| **검색이 틀리면 답도 틀린다** | 엉뚱한 조각을 찾아오면 LLM도 엉뚱하게 답함 (Garbage in, Garbage out) | 청킹·임베딩·K 튜닝, Re-ranking |
| **환각이 0이 되진 않는다** | 근거가 있어도 가끔 비틀어 말할 수 있음 | "자료만 근거" 지시 강화, 출처 노출, 검수 |
| **원본 품질에 의존** | 문서 자체가 틀렸거나 낡았으면 답도 그러함 | 문서 거버넌스(최신화·정리) |
| **표·이미지·복잡한 양식에 약함** | 파싱이 어려운 자료는 정확도 저하 | 표 전용 파서, OCR, 멀티모달 임베딩 |
| **보안·권한** | 잘못 구성하면 권한 없는 문서가 노출될 위험 | 메타데이터 기반 접근제어 |
| **평가가 필요** | "잘 되는지" 측정 없이는 개선 불가 | 질문-정답 세트로 정기 평가 |

> 📌 그래서 RAG는 **"한 번 만들고 끝"이 아니라, 문서 관리·튜닝·평가가 따라붙는 '운영하는 시스템'** 입니다.

### 부록 B. 용어 사전 (한눈에 복습)

| 용어 | 쉬운 설명 |
|------|-----------|
| **LLM** | ChatGPT 같은 거대 언어 모델. 그럴듯한 문장을 만드는 AI. |
| **RAG** | 답하기 전에 관련 문서를 **검색해 함께 넣어** 답하게 하는 방법. '오픈북 시험'. |
| **Hallucination(환각)** | AI가 사실 아닌 것을 그럴듯하게 지어내는 현상. |
| **Context Window** | AI가 한 번에 볼 수 있는 글의 최대 분량('책상 크기'). |
| **Token(토큰)** | 글을 잘게 나눈 조각. 분량·비용의 단위. |
| **Fine-tuning** | 모델을 추가 학습시켜 **행동·말투**를 바꾸는 것('재교육'). |
| **Ingestion** | 문서를 읽고 가공해 검색 가능한 형태로 창고에 넣는 준비 과정. |
| **Chunk / Chunking** | 문서 조각 / 문서를 조각내는 일('색인 카드 만들기'). |
| **Embedding(임베딩)** | 글의 **의미를 숫자 좌표(벡터)** 로 바꾸는 것. |
| **Vector(벡터)** | 숫자를 순서대로 나열한 목록. 의미의 '좌표'. |
| **Vector Database** | 벡터를 저장하고 비슷한 것을 빠르게 찾는 특수 DB. |
| **Retrieval(검색)** | 질문과 의미가 가까운 조각을 찾아오는 단계. |
| **Similarity Search** | 의미적으로 가장 가까운 벡터를 찾는 검색. |
| **Cosine Similarity** | 두 벡터의 '방향'이 얼마나 같은지로 유사도를 재는 법(1=비슷). |
| **Top-K** | 가장 비슷한 상위 K개만 골라 쓰는 것. |
| **Prompt Augmentation** | 검색한 자료+지시+질문을 한 프롬프트로 합치는 것('컨닝페이퍼'). |
| **ANN** | 근사 최근접 이웃 탐색. 빠른 유사도 검색의 핵심 기술. |
| **Hybrid Search** | 의미 검색 + 키워드 검색을 함께 쓰는 방식. |
| **Re-ranking** | 후보를 더 정밀히 다시 점수 매겨 순위를 정하는 단계. |
| **Graph RAG** | 지식 그래프(관계도)를 활용하는 RAG. |
| **Agentic RAG** | LLM이 스스로 계획·반복·도구사용을 하는 RAG. |

### 부록 C. "이것만은 기억하세요" — 최종 5문장 요약

1. **LLM은 우리 회사를 모르고, 모르면 지어낸다.** (환각·최신성·내부지식 문제)
2. **RAG는 '오픈북 시험'이다.** 답하기 전에 관련 문서를 찾아 함께 넣어 준다.
3. **준비 단계**(Chunking→Embedding→Vector DB)와 **응답 단계**(Retrieval→Augmentation→LLM)로 나뉜다.
4. **임베딩**으로 글의 의미를 좌표로 바꾸기 때문에, **글자가 달라도 뜻이 통하면** 찾아낸다.
5. **지식·최신성·출처가 중요하면 RAG, 말투·형식이면 Fine-tuning** — 보통은 **RAG부터** 시작한다.
