Section 1

1AI의 기억 문제 — 똑똑한데 왜 자꾸 틀릴까?

RAG를 이해하려면 먼저 ChatGPT 같은 LLM의 근본적 약점부터 알아야 합니다. RAG는 바로 이 약점을 메우려고 태어났기 때문입니다.

1-1. LLM은 정말 '문서를 기억'하고 있을까?

많은 사람이 ChatGPT를 "엄청나게 많은 문서를 외우고 있는 AI"라고 생각합니다. 절반만 맞습니다.

LLM(거대 언어 모델)은 방대한 글을 학습하며 "어떤 단어 다음에 어떤 단어가 올 확률이 높은가"를 익힌 확률 기반 문장 생성기입니다. 지식이 '신경망 가중치'에 녹아들긴 하지만, 도서관처럼 원문을 그대로 보관하고 찾아보는 것이 아닙니다.

💡 비유 — 시험 전날 벼락치기한 학생

LLM은 교과서 수백 권을 빠르게 훑고 시험장에 들어간 똑똑한 학생과 같습니다.

  • 전체 내용·흐름은 잘 기억합니다. → 일반 상식·요약·글쓰기는 잘함
  • "12쪽 셋째 줄에 정확히 뭐라 쓰여 있었지?"는 헷갈립니다. → 구체적 사실은 부정확
  • 게다가 교과서를 시험장에 못 가져갑니다. → 우리 회사 문서를 모름
📌 핵심

LLM은 "기억을 떠올려 그럴듯하게 말하는" 데 강하지, "특정 문서를 정확히 인용하는" 데 강한 것이 아닙니다.

1-2. Context Window — AI의 '책상 크기'

LLM은 질문하는 그 순간 입력창에 들어온 글자만 볼 수 있습니다. 한 번에 볼 수 있는 텍스트의 최대 분량을 컨텍스트 윈도우(Context Window)라고 하며, 토큰(token) 단위로 셉니다.

💡 토큰이란?

단어를 잘게 나눈 조각입니다. 영어는 단어 1개가 1~2토큰, 한국어는 글자 1~2개가 약 1토큰 정도. "출산휴가는 며칠인가요?"는 대략 10토큰 안팎입니다.

💡 비유 — 책상 크기

컨텍스트 윈도우는 AI가 한 번에 펼쳐 놓고 볼 수 있는 책상의 넓이입니다. 작으면 자료 몇 장, 크면 두꺼운 보고서도 올릴 수 있지만 무한대는 아닙니다.

최신 모델의 책상은 대략 12만 토큰(약 300쪽 책 한 권)부터, 큰 모델은 100만 토큰이 넘기도 합니다. 그래서 이런 질문이 나오죠. "그럼 회사 문서 전부를 매번 통째로 넣으면 되잖아?" 그렇게 못 하는 4가지 이유가 있습니다.

1 · 용량 한계
회사 문서는 수십만~수백만 쪽. 아무리 큰 책상도 다 못 올립니다.
2 · 비용
입력 토큰만큼 과금. 매번 책 한 권을 넣으면 요금 폭발.
3 · 속도
입력이 길수록 답변이 느려집니다.
4 · 중간 흘려듣기
너무 길면 앞·뒤만 보고 중간 핵심을 놓침(Lost in the Middle).
📌 핵심

"다 집어넣기"는 답이 아닙니다. 필요한 부분만 골라서 넣어야 합니다 — 이것이 RAG의 핵심 아이디어입니다.

1-3. Hallucination — AI가 '자신 있게 지어내는' 문제

환각(Hallucination)은 모델이 사실이 아닌 내용을 마치 사실처럼 그럴듯하고 자신감 있게 만들어내는 현상입니다. LLM은 본질적으로 "다음에 올 가장 그럴듯한 말"을 생성하는 기계라서, '그럴듯함'이 목표이지 '진실'이 목표가 아니기 때문입니다.

💡 비유 — 모른다고 말하기 싫은 신입사원

"작년 3분기 출장 규정이 어떻게 됐죠?"라고 물으면, 정확히 모르면서도 "아마… 1박당 8만 원 한도였던 걸로 기억합니다"라고 그럴듯하게 답하는 것과 같습니다.

실생활 예시: 존재하지 않는 논문·판례를 진짜처럼 인용하는 사고가 여러 번 보고됐습니다. 해외에서는 변호사가 AI가 만든 가짜 판례를 법원에 제출했다가 제재받은 사례도 있었습니다.

📌 핵심

환각은 "근거 없이 기억에 의존해 답할 때" 가장 심해집니다 → 근거 문서를 함께 주면 크게 줄어듭니다(RAG가 하는 일!).

1-4. 최신 정보 문제 — AI는 '입사일이 정해진 직원'

LLM은 특정 시점까지의 데이터로 학습됩니다(지식 컷오프). 컷오프 이후의 일은 태어날 때부터 모릅니다.

💡 비유 — 어느 날 인쇄가 멈춘 백과사전

인쇄 이후에 바뀐 법, 새 가격, 어제 공지된 정책은 그 안에 없습니다. 게다가 회사 내부 문서는 공개 인터넷에 없으니 컷오프와 무관하게 처음부터 모릅니다.

🧭 Section 1 한 줄 요약

LLM은 똑똑하지만 (1) 우리 회사 문서를 모르고 (2) 책상(컨텍스트)이 유한하며 (3) 모르면 지어내고 (4) 최신·내부 정보에 약하다. RAG는 이 넷을 "답하기 직전에 관련 문서를 찾아 함께 건네주는" 방식으로 한 번에 해결합니다.

Section 2

2RAG의 등장 — '오픈북 시험'이라는 발상

가장 먼저 떠오른 해법은 "우리 데이터로 AI를 다시 가르치자(Fine-tuning)"였습니다. 하지만 곧 한계에 부딪혔습니다.

2-1. Fine-tuning(미세조정)의 한계

Fine-tuning은 이미 학습된 LLM에 우리 데이터를 추가 학습시켜 모델의 두뇌(가중치) 자체를 조금 바꾸는 작업입니다.

💡 비유 — 직원을 '재교육'시키기

신입사원을 몇 달간 연수원에 보내 머릿속을 다시 세팅하는 것. 말투·업무방식을 바꾸는 데는 효과적입니다. 하지만 '지식을 최신으로 유지'하는 용도로는 적합하지 않습니다.

한계설명
느리고 비쌈규정 한 줄 바뀔 때마다 재학습. 연수원에 매번 보내는 셈.
'외운 것'이 부정확사실을 가중치에 욱여넣어도 여전히 '그럴듯하게' 떠올림 → 환각이 안 사라짐.
출처가 없음"어느 문서 몇 쪽"을 댈 수 없음. 규정·법률·금융에서 치명적.
기존 능력 손상무리한 학습은 원래 잘하던 걸 잊는 '파국적 망각'을 부를 수 있음.
보안·권한 처리"임원만 열람" 같은 권한별 차단을 모델 내부에 녹이기 어려움.
⚠️ 흔한 오해

"회사 데이터로 Fine-tuning하면 AI가 우리 회사를 다 알게 된다." → 실제로는 지식 주입에 비효율적입니다. Fine-tuning의 진짜 강점은 '무엇을 아는가'가 아니라 '어떻게 행동·표현하는가'를 바꾸는 것입니다.

2-2. 기업이 진짜 원하는 것 · 2-3. 지식 최신성

"우리 회사의 수많은 내부 문서(규정·매뉴얼·계약서·회의록·위키)를 근거로, AI가 정확하고, 출처가 분명하며, 항상 최신인 답을 해 줬으면 좋겠다."

문서는 매일 추가·수정되고, 부서별 권한이 다르며, "근거가 어디냐"를 반드시 댈 수 있어야 합니다. 또 법 개정·가격 변동처럼 세상은 계속 바뀝니다. 모델을 통째로 다시 학습시키지 않고도 지식을 즉시 갱신할 방법이 필요합니다.

2-4. 발상의 전환 — "외우게 하지 말고, 찾아보게 하자"

💡 RAG = 폐쇄형 시험을 '오픈북 시험'으로 바꾸기
  • 기존 LLM: 책 없이 머리로만 푸는 폐쇄형 시험 → 헷갈리면 지어냄(환각).
  • Fine-tuning: 시험 전 학생을 재교육 → 비싸고 느리고, 새 내용 반영이 어려움.
  • RAG: 시험을 오픈북으로! 문제를 받으면 → 관련 페이지를 즉시 찾아 → 보면서 답을 작성 📖
📌 RAG의 정의

RAG(Retrieval-Augmented Generation, 검색 증강 생성)란, LLM이 답을 생성하기 직전에 질문과 관련된 문서를 외부 지식 창고에서 검색(Retrieval)해 와서, 그 내용을 함께 넣어(Augmented) 답을 만들게(Generation) 하는 방법입니다.

R
Retrieval · 검색
질문과 관련된 문서를 외부 창고에서 찾아온다.
A
Augmented · 증강
찾아온 문서를 질문과 함께 LLM 입력에 더한다.
G
Generation · 생성
LLM이 그 근거를 바탕으로 답을 만든다.
항목RAG가 주는 이점
정확성근거 문서를 보고 답하므로 환각이 크게 감소
출처"어느 문서에서 나왔는지" 인용 가능
최신성모델을 안 건드리고 문서만 갱신하면 즉시 반영
비용통째로 재학습할 필요 없음
보안사용자 권한에 따라 검색 대상 문서를 제어 가능
🧭 Section 2 한 줄 요약

AI를 매번 재교육(Fine-tuning)하는 대신, 답하기 직전에 관련 문서를 찾아 쥐여주는 '오픈북 시험'(RAG)으로 바꾸면 — 정확성·출처·최신성·비용 문제가 한꺼번에 풀린다.

Section 3

RAG는 8단계로 이뤄집니다. 먼저 전체 그림을 새기고, 다음 단원부터 하나씩 자세히 봅니다.

🔢

🗄️

그림 3-1. RAG 전체 흐름 — 위쪽 파란 구역은 '미리 준비', 아래쪽 초록 구역은 '질문 시 실시간'

3-1. 가장 중요한 구분 — '준비'와 '응답'은 다른 시점이다

📌 이것만 잡아도 절반은 이해

위 흐름은 사실 두 개의 다른 시점으로 나뉩니다.

  • ① 준비 단계 (Indexing) — Document → Chunking → Embedding → Vector DB. 도서관에 새 책이 들어오면 사서가 미리 분류해 서가에 꽂아 두는 작업. 질문 전에 끝냄.
  • ② 응답 단계 (Retrieval & Generation) — 질문 → Retrieval → Augmentation → LLM → Answer. 이용자가 질문하면 사서가 관련 책을 찾아다 주고, 보고 답하는 작업. 질문할 때마다 실행.
💡 도서관 비유로 전체 묶기
  • Chunking = 책을 주제별 색인 카드로 정리
  • Embedding = 각 카드에 '의미 주소' 부여
  • Vector DB = 카드들을 의미 주소순으로 보관하는 특별한 서가
  • Retrieval = 사서가 관련 카드를 찾아다 줌
  • Augmentation = 찾은 카드를 질문과 함께 똑똑한 직원에게 전달
  • LLM = 그 자료를 읽고 사람이 이해할 문장으로 답을 작성

3-2. 각 단계 한 줄 설명

#단계한 줄 설명비유
1Document활용할 원본 자료도서관에 들어온 책
2Chunking긴 문서를 적당한 조각으로 분할책을 색인 카드로 분리
3Embedding조각의 의미를 숫자 벡터로 변환카드에 의미 주소 부여
4Vector DB벡터를 저장·색인의미순 정리 서가
5Retrieval질문과 가까운 조각 Top-K 검색사서가 관련 카드 찾기
6Augmentation검색 조각을 질문과 합쳐 프롬프트화직원에게 자료+질문 전달
7LLM자료를 근거로 답변 생성직원이 답안 작성

Section 5

5Chunking — 문서를 어떻게 쪼갤 것인가

5-1. Chunking이 왜 필요한가?

청크(chunk)는 '덩어리·조각'. 긴 문서를 통째로 두지 않고 굳이 쪼개는 이유는 셋입니다.

  1. 검색 정확도: 100쪽을 하나의 좌표로 만들면 주제가 뒤섞여 흐릿. 작게 쪼개야 각 조각이 분명한 한 주제를 가짐.
  2. 컨텍스트 절약: 책상(컨텍스트)은 유한 → 필요한 그 문단 몇 개만 넣어야 효율적.
  3. 정밀한 출처: "7쪽 휴가 조항에서 나왔다"처럼 콕 집어 근거 제시 가능.
💡 비유 — 책 vs 색인 카드

"출산휴가" 한 줄을 찾는데 1,000쪽 책을 통째로 받으면 곤란합니다. "출산휴가 = 90일"이라 적힌 색인 카드 한 장이 훨씬 유용하죠.

⚠️ 너무 잘게도, 너무 크게도 문제
  • 너무 작게(한 문장): 문맥 끊김 — "90일"만 남으면 무엇이 90일인지 모름
  • 너무 크게(한 챕터): 여러 주제 혼합 → 검색 부정확·컨텍스트 낭비

5-2. 네 가지 대표 방식

① Fixed — 고정 크기

의미와 무관하게 N글자마다 싹둑.

출산휴가는 90✂일입니다. 다태

단어·의미 중간이 잘림

② Recursive — 재귀적 ⭐

문단→문장→단어 순 자연 경계 우선.

문단▸문장.▸단어

구조 보존, 가장 무난

③ Semantic — 의미 기반

주제가 바뀌는 지점에서 자름.

출산휴가…┊연차휴가…

품질 최상, 단 느리고 비쌈

앞 조각 끝을 다음 조각에 겹침.

경계 문맥 보존(보완용)

5-3. 비교표

방식동작 원리장점단점사용 사례

Python · Chunking
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 한 조각 최대 길이(문자)
    chunk_overlap=50,      # 인접 조각 간 겹침 → 경계 문맥 보존
    separators=["\n\n", "\n", ". ", " ", ""],  # 위에서부터 시도
)
chunks = splitter.split_text(long_text)

# 슬라이딩 윈도우(겹침)를 직접 구현하면 이런 원리:
def sliding(text, size=500, overlap=50):
    step = size - overlap
    return [text[i:i+size] for i in range(0, len(text), step)]

separators가 '문단→문장→단어' 우선순위입니다. 토큰 기준으로 자르려면 RecursiveCharacterTextSplitter.from_tiktoken_encoder(...)를 씁니다.

Section 6

6Embedding — 의미를 '숫자 좌표'로 바꾸기

RAG에서 가장 중요하면서도 가장 신기한 부분입니다. 천천히 갑니다.

6-1. 왜 단어를 '숫자'로?

컴퓨터에게 "고양이"와 "야옹이"는 그냥 다른 글자 덩어리일 뿐, 비슷하다는 걸 모릅니다. 컴퓨터가 잘 다루는 건 숫자뿐이죠. 그래서 단어·문장의 '의미'를 숫자(좌표)로 표현한 것이 임베딩(Embedding) 벡터입니다.

📌 임베딩의 핵심 원리

'의미가 비슷한 것'은 '숫자 좌표도 가깝게' 만들어진다. (벡터 = 숫자를 나열한 목록, 예 [0.21, -0.88, 0.05, …] — 실제론 수백~수천 개)

6-2. 벡터 공간 — '의미의 지도'

💡 비유 — 지도 위의 도시

지도에서 '서울'과 '인천'은 가깝고 '서울'과 '부산'은 멉니다. 위도·경도라는 숫자로 위치를 표현했기 때문. 임베딩도 똑같되, 숫자가 수백~수천 개이고 거리는 '의미의 가까움'입니다.

6-3. 직접 보기 — 고양이 / 강아지 / 자동차

설명을 위해 차원을 딱 2개(실제론 수백~수천)로 줄인 가상의 좌표입니다.

그림 6-1. 의미의 지도 — 고양이·강아지는 붙어 있고(가까움), 자동차는 멀리 떨어져 있다
  • 고양이 ↔ 강아지: 좌표가 거의 붙음 — 둘 다 '털 달린 반려동물'이라 의미가 비슷.
  • 고양이 ↔ 자동차: 멀리 떨어짐 — '동물'과 '기계'는 의미가 전혀 다름.
📌 RAG가 작동하는 비결

"반려묘"로 검색해도 글자가 다른 "고양이" 문서를 찾아냅니다. 글자가 아니라 '의미의 위치'로 찾기 때문 — 이것이 의미 기반 검색(semantic search).

6-4. 키워드 검색과의 결정적 차이

질문 → 문서 표현키워드 검색 (글자 일치)의미 검색 (임베딩)
"출산휴가 며칠?" → "출산전후휴가 기간"✘ 글자 달라 못 찾음✔ 의미 같아 찾음
"노트북 안 켜져요" → "전원이 안 들어올 때"✘ 못 찾음✔ 찾음
"퇴사 절차" → "이직 시 인수인계"✘ 못 찾음✔ 관련 문서로 찾음
💡 비유 — 융통성 없는 사서 vs 베테랑 사서

키워드 검색은 "출산휴가"라고 정확히 말해야만 책을 주는 융통성 없는 사서. 의미 검색은 "출산 후 쉬는 거요"라고 해도 "아, 출산전후휴가요!" 하고 찰떡같이 알아듣는 베테랑 사서.

6-5. 임베딩은 누가 만드나?

전용 AI인 임베딩 모델이 자동으로 만듭니다(2025~2026 대표):

  • OpenAI text-embedding-3-small / large — 간편·고성능, 가장 널리 쓰임 (1,536 / 3,072차원)
  • Cohere embed-v4 — 다국어(한국어 포함) 강점, 기업용
  • BGE-M3 (오픈소스) — 무료 자체 운영 가능, 100+ 언어
⚠️ 중요한 규칙

문서를 임베딩한 모델과 질문을 임베딩하는 모델은 반드시 같아야 합니다. 서로 다른 모델은 서로 다른 '의미 지도'를 쓰기 때문에, 같은 지도 위에서 비교해야 거리가 맞습니다.

from langchain_openai import OpenAIEmbeddings

emb = OpenAIEmbeddings(model="text-embedding-3-small")
v_cat = np.array(emb.embed_query("고양이"))
print(v_cat.shape)        # (1536,) ← 의미를 담은 1536차원 벡터

def cos(a, b):            # 코사인 유사도 (@ = 내적)
    return a @ b / (np.linalg.norm(a) * np.linalg.norm(b))

v_dog = np.array(emb.embed_query("강아지"))
v_car = np.array(emb.embed_query("자동차"))
print(round(cos(v_cat, v_dog), 3))   # 더 높음 → 의미가 가깝다
print(round(cos(v_cat, v_car), 3))   # 더 낮음 → 의미가 멀다

문서 전체를 한 번에 벡터화할 땐 emb.embed_documents([chunk1, chunk2, ...])를 씁니다(배치 처리로 빠름). 반환값은 그냥 실수 리스트라 바로 np.array로 다룰 수 있습니다.

7-5. 인덱스 구조와 ANN — '어떻게' 빨리 찾나

저장된 벡터를 질문마다 전부 비교하면 느립니다(§8의 numpy 전수 비교). 그래서 벡터 DB는 인덱스를 미리 만들어 둡니다. 대표적으로 세 가지:

인덱스방식속도정확도(재현율)메모리적합 규모
Flat전수 비교(brute-force)느림100% 정확낮음~수만
IVF클러스터로 나눠 일부만 탐색빠름높음(근사)중간수십만~수백만
HNSW다층 그래프를 따라 탐색매우 빠름매우 높음(근사)높음수백만~수십억

요즘 가장 널리 쓰이는 건 HNSW(많은 벡터 DB의 기본값)입니다. DNN을 아는 청중에게 익숙한 '그래프 탐색'으로 작동합니다 — 위층의 성긴 그래프에서 대충 가까운 곳으로 점프한 뒤, 아래층의 촘촘한 그래프에서 정밀하게 좁혀 갑니다.

그림 7-2. HNSW — 성긴 상위 층에서 진입해 점점 촘촘한 하위 층으로 내려가며 최근접 이웃을 찾는다(그리디 탐색)
🐍 파이썬으로는 — faiss로 인덱스 만들기
Python · faiss Index
import faiss, numpy as np

d = 1536                                    # 벡터 차원
xb = np.array(chunk_vecs, dtype="float32")  # (N, d)

# (1) Flat: 전수 비교 — 가장 정확, 느림
index = faiss.IndexFlatIP(d)                # IP=내적, 정규화하면 코사인
faiss.normalize_L2(xb); index.add(xb)

# (2) IVF: 클러스터로 나눠 일부만 탐색 — 빠름
quant = faiss.IndexFlatIP(d)
index = faiss.IndexIVFFlat(quant, d, 100)   # nlist=100 클러스터
index.train(xb); index.add(xb); index.nprobe = 8   # 8개 클러스터만 탐색

# (3) HNSW: 그래프 기반 — 대규모에서 빠르고 정확(많은 DB의 기본)
index = faiss.IndexHNSWFlat(d, 32)          # M=32: 노드당 이웃 수
index.add(xb)

# 검색: q=(1, d) 질문 벡터 → 거리 D, 인덱스 I
faiss.normalize_L2(q)
D, I = index.search(q, k=3)
📌 거리 척도 한 가지만

L2(유클리드)는 작을수록, IP(내적)·코사인은 클수록 유사합니다. 코사인을 쓰고 싶으면 벡터를 L2 정규화한 뒤 IP 인덱스를 쓰면 됩니다(위 코드의 normalize_L2). 정규화된 벡터에서는 IP = 코사인.

Python · Retrieval (Top-K)
import numpy as np

# chunk_vecs: 저장된 청크 벡터 행렬 (N, 1536), q: 질문 벡터 (1536,)
q = np.array(emb.embed_query("출산휴가는 며칠인가요?"))

def normalize(x):
    return x / np.linalg.norm(x, axis=-1, keepdims=True)

sims = normalize(chunk_vecs) @ normalize(q)   # 코사인 유사도 (N,)
k = 3
topk = np.argsort(sims)[::-1][:k]             # 상위 k개 인덱스
print(topk, sims[topk])                        # 예: [2 ...] [0.95 ...]

# LangChain이면 이 모든 게 한 줄:
results = vectorstore.similarity_search_with_score("출산휴가는 며칠인가요?", k=3)

위 numpy 버전은 '전수 비교(brute-force)'라 작은 데이터엔 충분하지만, 수백만 개부터는 §7의 ANN 인덱스가 이 계산을 근사·가속합니다.

Prompt Augmentation은 검색 근거 + 행동 지시 + 사용자 질문을 한 장의 프롬프트로 합쳐 LLM에 건네는 단계. 이 '컨닝페이퍼' 덕분에 근거 있고 출처 있는 답이 나온다.

11-2. Graph RAG 깊이 보기

요청이 많은 주제라 따로 자세히 봅니다. 일반 RAG가 '텍스트 조각'을 검색한다면, Graph RAG는 '엔티티(개체)와 관계'를 검색합니다.

개념 — 지식 그래프 = 노드(엔티티) + 엣지(관계)

문서에서 개체와 그 사이 관계를 뽑아 triple 형태 (주어, 관계, 목적어)로 저장합니다. 예: (김대리, 소속, 인사팀).

그림 11-1. 지식 그래프 — "김대리가 속한 부서가 담당하는 정책의 휴가, 그 근거 법은?" 같은 질문에 엣지를 따라 multi-hop 추론으로 답한다

왜 필요한가 — 벡터 RAG의 한계

⚠️ 일반 RAG가 약한 지점
  • 흩어진 사실 연결: 답이 여러 문서에 조각조각 흩어져 있으면, 청크 유사도만으론 다 모으기 어렵다.
  • 다단계(multi-hop) 추론: "A의 상위 조직이 담당하는 정책의 근거 법" 같은 관계 추적이 약하다.
  • 전역 요약: "이 문서 전체에서 가장 중요한 리스크 3가지" 같은 거시 질문에 약하다.

Graph RAG는 관계를 명시적 그래프로 갖고 있어, 이런 연결·추론·요약 질문에서 강합니다.

저장 형식 + 코드 — property graph / triple

그래프 DB는 보통 property graph(노드·관계에 속성 부여) 또는 RDF triple(주어-술어-목적어) 형식으로 저장합니다. 가장 대중적인 Neo4j의 질의어 Cypher로 보면:

Cypher · Neo4j
// 노드와 관계 생성 (property graph)
CREATE (kim:Employee {name:'김대리'})-[:BELONGS_TO]->(hr:Dept {name:'인사팀'}),
       (hr)-[:OWNS]->(p:Policy {name:'복지정책'}),
       (p)-[:INCLUDES]->(:Leave {name:'출산휴가', days:90});

// multi-hop 질의: 김대리 부서가 담당하는 정책의 휴가는?
MATCH (:Employee {name:'김대리'})-[:BELONGS_TO]->(:Dept)
      -[:OWNS]->(:Policy)-[:INCLUDES]->(l:Leave)
RETURN l.name, l.days;   // → "출산휴가", 90

파이썬 파이프라인에서는 LangChain LLMGraphTransformer가 문서에서 위 같은 노드·관계를 자동 추출해 그래프 DB에 적재합니다. 가벼운 실험은 networkx + LLM으로도 가능합니다.

대표 그래프 DB · 구축 도구

  • 그래프 DB: Neo4j(가장 대중적) · Amazon Neptune · TigerGraph · ArangoDB
  • 벡터+그래프 겸용: Neo4j(벡터 인덱스 내장) · Weaviate
  • 구축 프레임워크: Microsoft GraphRAG · LangChain(LLMGraphTransformer) · LlamaIndex(KnowledgeGraphIndex)
🔧 실무 선택

대부분은 벡터 RAG로 충분합니다. 관계·다단계 추론·전역 요약이 핵심인 도메인(법률·규정·조직·제조 BOM 등)에서 벡터 + 그래프 하이브리드로 가는 것이 현실적입니다. 비용이 부담이면 LazyGraphRAG·LightRAG 같은 경량 변형부터.

"먼저 RAG로 시작하라." 대부분의 기업 요구(사내 지식·최신성·출처)는 RAG로 해결됩니다. 말투·형식이 부족하면 그때 Fine-tuning을 추가하세요.

⚠️ 다시 강조

Fine-tuning은 '지식 주입기'가 아닙니다. 자주 바뀌는 사실을 넣으려는 시도는 비싸고, 환각도 못 막고, 출처도 못 댑니다 — 그 일은 RAG의 몫.

🧭 Section 13 한 줄 요약

지식·최신성·출처가 중요하면 RAG, 말투·형식·행동을 바꾸려면 Fine-tuning. 둘은 보완 관계이며 대부분 RAG부터 시작하는 것이 정답.

예제 1. 최소 구현 — OpenAI + LangChain + FAISS

"RAG가 정말 이렇게 짧아?" 싶을 만큼 핵심만 담은 버전입니다.

Python · 예제 1 (최소 구현)
# pip install langchain langchain-openai langchain-community faiss-cpu

from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# 0) 우리 회사 문서 (지금은 간단히 문자열 몇 개로 대신)
texts = [
    "출산전후휴가는 출산 전후를 통하여 총 90일이 부여된다. 다태아는 120일이다.",
    "연차휴가는 1년간 80% 이상 출근 시 15일이 부여된다.",
    "경조사 휴가 중 본인 결혼은 5일, 배우자 출산은 10일이다.",
]

# 1) 임베딩 모델 준비 + 2) 벡터DB(FAISS)에 저장
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_texts(texts, embeddings)

# 3) 검색기: 질문과 가장 비슷한 조각 2개를 찾도록 설정
retriever = vectorstore.as_retriever(search_kwargs={"k": 2})

# 4) 프롬프트: '자료를 보고 답하라'는 지시 + 자료 자리 + 질문 자리
prompt = ChatPromptTemplate.from_template(
    "아래 자료만 근거로 질문에 답하세요. 없으면 모른다고 하세요.\n\n"
    "자료:\n{context}\n\n질문: {question}"
)

# 5) 답변 생성 LLM
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 검색된 조각들을 하나의 글로 합치는 도우미
def format_docs(docs):
    return "\n".join(d.page_content for d in docs)

# 6) 전체 RAG 흐름 연결: 검색 → 프롬프트 → LLM → 텍스트
chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 7) 질문!
print(chain.invoke("출산휴가는 며칠인가요?"))
# 예상 출력: 출산전후휴가는 총 90일이며, 다태아의 경우 120일입니다.
💡 이 코드가 하는 일 (한 문장씩)
  • texts : 회사 규정을 문자열로 준비 = Document/Chunk
  • OpenAIEmbeddings : 글을 의미 좌표(벡터)로 바꾸는 변환기 = Embedding
  • FAISS.from_texts : 좌표들을 검색 가능한 창고에 저장 = Vector DB/Indexing
  • as_retriever(k=2) : "가장 비슷한 2조각을 찾아 줘" 검색기 = Retrieval
  • prompt : "이 자료만 보고 답해" 지시문 틀 = Prompt Augmentation
  • llm : 답 문장을 써 주는 AI = LLM (temperature=0 = 창의적 변형을 줄여 일관·안정적 답, 사실 답변에 적합)
  • chain : 위 단계를 검색→자료합치기→프롬프트→LLM 순서로 연결한 컨베이어 벨트

예제 2. 실무 스타일 — 진짜 PDF로 RAG 구축

실제 업무처럼 PDF를 읽고 정식 단계(Loader → Splitter → Embedding → Vector Store → Retriever → PromptTemplate)를 모두 갖춘 버전입니다.

Python · 예제 2 (실무 스타일)
# pip install langchain langchain-openai langchain-community faiss-cpu pypdf

from langchain_community.document_loaders import PyPDFLoader          # Loader
from langchain_text_splitters import RecursiveCharacterTextSplitter  # Splitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI            # Embedding & LLM
from langchain_community.vectorstores import FAISS                    # Vector Store
from langchain_core.prompts import ChatPromptTemplate                # PromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# ── [준비 단계: 문서를 검색 가능하게 만들기] ──────────────────

# 1) Loader: PDF를 페이지 단위로 읽어 들인다
loader = PyPDFLoader("사내복지규정.pdf")
documents = loader.load()

# 2) Splitter: 긴 페이지를 500자(50자 겹침)로 쪼갠다 (Recursive)
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 한 조각의 최대 길이
    chunk_overlap=50,      # 조각 사이 겹치는 길이(문맥 보존)
)
chunks = splitter.split_documents(documents)

# 3) Embedding + 4) Vector Store: 조각을 벡터로 바꿔 FAISS에 저장
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(chunks, embeddings)
# vectorstore.save_local("faiss_index")  # 다음엔 재계산 없이 불러오기

# ── [응답 단계: 질문에 답하기] ───────────────────────────────

# 5) Retriever: 질문과 가장 비슷한 조각 4개를 가져오도록 설정
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# 6) PromptTemplate: 역할·규칙·자료·질문을 한 장으로 구성
prompt = ChatPromptTemplate.from_template(
    """당신은 사내 규정 안내 도우미입니다.
아래 [참고 자료]에 있는 내용만 근거로 답변하세요.
자료에 없으면 추측하지 말고 "규정에서 찾을 수 없습니다"라고 답하세요.

[참고 자료]
{context}

[질문]
{question}

[답변]"""
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# 전체 파이프라인 연결 (검색 → 프롬프트 → LLM → 문자열)
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

answer = rag_chain.invoke("출산휴가는 며칠인가요?")
print(answer)
💡 이 코드가 하는 일 (단계별)
  • 1) Loader — PDF를 열어 글자를 읽음. Word·웹·노션은 다른 로더로 교체. = Document Loading
  • 2) Splitter — 500자, 50자 겹침으로 쪼갬. 겹침은 경계 문맥 보존 장치. = Chunking
  • 3) Embedding — 각 조각을 의미 좌표(벡터)로. = Embedding
  • 4) Vector Store — 벡터를 검색 가능한 창고에 저장. save_local로 저장해 두면 재계산 없이 재사용. = Indexing
  • 5) Retriever — 질문 시 가장 비슷한 4조각을 찾음. = Retrieval
  • 6) PromptTemplate — "자료만 근거로, 없으면 모른다고"라는 규칙 틀. 답변 품질을 크게 좌우. = Prompt Augmentation
  • rag_chain — 모든 단계를 하나로 연결. invoke("질문") 한 줄로 검색→근거 결합→답변 생성이 자동 실행.
🔧 실무에서 더 신경 쓰는 것들
  • 문서 변경 시 해당 부분만 다시 인제스천(증분 업데이트)
  • 답변에 실제 출처(파일명·페이지) 노출 → 신뢰도↑ (메타데이터)
  • 정답률 낮으면 → 청크 크기·K 조정, Re-ranking·Hybrid Search(§11) 추가
  • 평가(Evaluation): 예상 질문-정답 세트로 정답률 정기 측정
  • 보안: 사용자 권한에 따라 검색 대상 문서 제한(메타데이터 필터링)

예제 3. 프레임워크 없이 — numpy로 보는 RAG의 본질

LangChain을 걷어내고 openai + numpy만으로 짜 보면, 앞서 본 단위 코드(임베딩 → 코사인 → Top-K → 프롬프트 → LLM)가 하나의 흐름임이 드러납니다. RAG에 마법은 없습니다.

Python · 예제 3 (numpy from scratch)
# pip install openai numpy
import numpy as np
from openai import OpenAI
client = OpenAI()                       # OPENAI_API_KEY 환경변수 사용

docs = [
    "출산전후휴가는 출산 전후 통산 90일(다태아 120일).",
    "연차휴가는 1년 80% 이상 출근 시 15일.",
    "본인 결혼 경조휴가는 5일.",
]

def embed(texts):                       # 텍스트 → 벡터 (Embedding)
    r = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return np.array([d.embedding for d in r.data])

doc_vecs = embed(docs)                  # (3, 1536) ← 준비 단계(인덱싱)

# --- 응답 단계 ---
q = "출산휴가는 며칠인가요?"
qv = embed([q])[0]
def cos(a, b): return a @ b / (np.linalg.norm(a) * np.linalg.norm(b))
sims = [cos(qv, dv) for dv in doc_vecs]          # 코사인 유사도 (Retrieval)
top = int(np.argmax(sims))                       # Top-1 선택
context = docs[top]

prompt = f"자료만 근거로 답하세요.\n자료: {context}\n질문: {q}"   # (Augmentation)
ans = client.chat.completions.create(            # (Generation)
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": prompt}],
)
print(ans.choices[0].message.content)            # → "총 90일 …"
📌 이 30줄이 RAG의 전부

프레임워크(LangChain 등)는 이 흐름에 로더·청킹·벡터DB·재시도·평가를 얹어 규모와 운영을 편하게 해 줄 뿐, 본질은 위 임베딩 → 코사인 → Top-K → 프롬프트 → LLM입니다.

🧭 Section 14 한 줄 요약

코드로 본 RAG도 결국 Loader → Splitter → Embedding → Vector Store → Retriever → Prompt → LLM의 한 줄 흐름. LangChain은 이 조각들을 레고처럼 끼워 맞추는 도구일 뿐이다.

C. "이것만은 기억하세요" — 최종 5문장

🧭 핵심 5문장
  1. LLM은 우리 회사를 모르고, 모르면 지어낸다. (환각·최신성·내부지식 문제)
  2. RAG는 '오픈북 시험'이다. 답하기 전에 관련 문서를 찾아 함께 넣어 준다.
  3. 준비 단계(Chunking→Embedding→Vector DB)와 응답 단계(Retrieval→Augmentation→LLM)로 나뉜다.
  4. 임베딩으로 의미를 좌표로 바꾸기에, 글자가 달라도 뜻이 통하면 찾아낸다.
  5. 지식·최신성·출처는 RAG, 말투·형식은 Fine-tuning — 보통 RAG부터 시작한다.
